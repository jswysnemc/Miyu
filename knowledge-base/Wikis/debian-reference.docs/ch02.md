### []{#_debian_package_management_system}2.1.1. Debian 软件包管理 {#debian-软件包管理 .title}

</div>

</div>
:::::

[Debian](https://www.debian.org){.ulink} 是一个志愿者组织，它建立[**一致的**]{.strong}自由软件的预编译二进制包并从档案库中分发它们。

[许多远程镜像站](https://www.debian.org/mirror/){.ulink}提供了 HTTP 和 FTP 的方式来访问 [Debian 档案库](http://deb.debian.org/debian/){.ulink}。也可以使用 [CD-ROM/DVD](https://www.debian.org/CD/){.ulink} 。

目前 Debian 的软件包管理系统是 [高级软件包工具 (APT)](https://en.wikipedia.org/wiki/APT_(software)){.ulink}，它能够使用所有这些资源。

Debian 软件包管理系统，[**当使用适当时**]{.strong}，可以让用户从档案库安装[**统一设置的二进制软件包**]{.strong} 到系统中。现在，有 74874 个可用于 amd64 架构的软件包。

Debian 软件包管理系统有丰富的历史，有许多可供选择的前端用户程序和后端访问方式。现在，我们推荐下列的这些。

::: itemizedlist
-   `apt`{.literal}(8) 用于所有的交互式命令行操作，包含软件包的安装、移除和版本升级。

-   `apt-get`{.literal}(8) 用于从脚本中调用 Debian 软件包管理系统。它在 `apt`{.literal} 不可用时也可作为一个备选选项（常见于较旧的 Debian 系统）。

-   `aptitude`{.literal}(8) 使用一个交互式的文本界面来管理已安装的软件包和搜索可用的软件包。
:::

### []{#_package_configuration}2.1.2. 软件包配置 {#软件包配置 .title}

</div>

</div>
:::::

下面是 Debian 系统软件包配置的一些要点。

::: itemizedlist
-   对于现代桌面环境来说，在软件包配置修改和软件包升级后重启系统是确保系统正常运行的好主意。

-   Debian 尊重系统管理员的手动配置。换句话说，软件包配置系统不会为了方便而去更改那些配置。

-   每个软件包都带有自己的配置脚本，它使用标准用户接口 `debconf`{.literal}(7) 来帮助软件包初始化安装过程。

-   Debian 开发者通过软件包配置脚本，尽力使你能有一个完美的升级体验。

-   系统管理员可以使用软件包工具的全部功能。但在默认的安装中会禁用那些具有安全风险的。

-   如果你手动激活了一些具有安全隐患的服务，你有责任遏制风险。

-   高深的配置可以由系统管理员手动启用。这可能会对用于系统配置的通用流行帮助程序造成干扰。
:::

### []{#_basic_precautions}2.1.3. 基本的注意事项 {#基本的注意事项 .title}

</div>

</div>
:::::

::: {.warning style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[警告\]](images/warning.png)  警告
                                    不要从任何的混合套件中安装软件包。它可能会打破软件包的一致性，这需要你要深厚的系统管理知识，例如 [ABI](https://zh.wikipedia.org/wiki/应用程序二进制接口){.ulink} 编译器、[库](https://zh.wikipedia.org/wiki/Library_(computing)){.ulink}版本和解释器特性等等。
  --------------------------------- ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

Debian 系统管理员中的[新手](https://zh.wikipedia.org/wiki/Newbie){.ulink}应该保持在只进行安全更新的 [**`stable`{.literal}**]{.strong} 版本。直到你十分了解 Debian 系统前，你应当遵循下列的预防措施。

::: itemizedlist
-   在 [**源文件**]{.strong} 中不要包含 [**`testing`{.literal}**]{.strong} 或 [**`unstable`{.literal}**]{.strong} 。

-   在 [**源文件**]{.strong} 里不要在标准的 Debian 中混合使用其它非 Debian 的档案库，例如 Ubuntu 。

-   不要建立 "`/etc/apt/preferences`{.literal}" 。

-   不了解会造成的全部影响，就不要通过配置文件改变软件包管理工具的默认行为。

-   不要使用 "`dpkg -i `{.literal}[*`random_package`{.literal}*]{.emphasis}" 安装任何软件包。

-   绝不使用 "`dpkg --force-all -i `{.literal}[*`random_package`{.literal}*]{.emphasis}" 安装任何软件包。

-   不要删除或修改 "`/var/lib/dpkg/`{.literal}" 中的文件。

-   不要让从源码直接安装的程序覆盖系统文件。

    ::: itemizedlist
    -   如果需要的话，将它们安装到 "`/usr/local`{.literal}" 或 "`/opt`{.literal}" 中。
    :::
:::

对 Debian 软件包管理系统，违背上面的预防措施，会导致不兼容影响，可能会使你的系统无法使用。

运行关键任务服务器的严谨的 Debian 系统管理员，应该使用额外的预防措施。

::: itemizedlist
-   没有在安全的条件下使用你特定的配置进行彻底地测试，就不要从 Debian 安装任何软件包（包含安全更新）。

    ::: itemizedlist
    -   你作为系统管理员要对你的系统负责到底。

    -   Debian 系统长久的稳定史并无法保证什么。
    :::
:::

                                    对于你的[**生产服务器**]{.strong}，建议使用带有安全更新的 `stable`{.literal} 套件。对于你只进行有限管理的桌面 PC 也是同样如此。
  --------------------------------- ---------------------------------------------------------------------------------------------------------------------------------
:::

尽管我在上面进行了警告，我知道本文档的许多读者希望可以运行更新的 `testing`{.literal} 或 `unstable`{.literal} 版。

[菩萨](https://zh.wikipedia.org/wiki/Bodhi){.ulink}使用下面的内容拯救一个人，使他从挣扎于持续升级[地狱](https://zh.wikipedia.org/wiki/Naraka){.ulink}的[因果报应](https://zh.wikipedia.org/wiki/Karma){.ulink}中脱困，并让他达到 Debian 的[极乐世界](https://zh.wikipedia.org/wiki/Nirvana){.ulink}。

这个列表面向 [**自己管理的**]{.strong} 桌面环境。

::: itemizedlist
-   使用 `testing`{.literal} 版，实际上，它是自动滚动发布的，由 Debian 档案库的 QA 质量架构来管理，比如：[Debian 持续集成](https://ci.debian.net/){.ulink}、[只上传源代码实践](https://wiki.debian.org/SourceOnlyUpload){.ulink} 和 [库转换跟踪](https://wiki.debian.org/Teams/ReleaseTeam/TransitionTracker){.ulink}。在 `testing`{.literal} 版中的软件包被更新得足够频繁来提供全部最新的特性。

-   在[**源列表**]{.strong}里面设置 `testing`{.literal} 版相应的代码名为套件名（在 `trixie`{.literal}-作为-`stable`{.literal} 版的发布周期时，是\"`forky`{.literal}\"）。

-   大概在主版本发布一个月后，仅仅在你自己评估了形势后，才手动更新 [**源文件**]{.strong} 里的这个代码名到新的版本号。对于这个更新，Debian 用户和开发者邮件列表也是好的信息来源。
:::

使用 `unstable`{.literal} 版是不推荐的。`unstable`{.literal} 版对开发者 [**调试软件包合适**]{.strong}，但对普通的桌面使用而言，会有使你暴露在不必要的风险中的倾向。 尽管 Debian 系统的 `unstable`{.literal} 版在大多数时候看起来都非常稳定，但会有一些软件包问题，并且它们中的一部分是不容易解决的。

这里有一些基本预防措施意见，确保简单快速地从 Debian 软件包的 bug 中恢复。

::: itemizedlist
-   使用下面的 [第 3.2 节 "应急救援系统"](ch03.zh-cn.html#_rescue_system "3.2. 应急救援系统"){.xref} 制作紧急救援系统。

-   通过将 Debian 系统的 `stable`{.literal} 版安装到另一个分区，可以使系统能够进行[**双启动**]{.strong}

-   考虑安装 `apt-listbugs`{.literal} ，这可以在升级之前检查 [Debian Bug 跟踪系统（BTS）](https://www.debian.org/Bugs/){.ulink} 的信息

-   对软件包系统的基础设施有足够的了解来解决问题
:::

让我们从系统用户的角度来看看 [Debian 档案库](http://deb.debian.org/debian/){.ulink}。

对于系统用户，是使用 APT 系统来访问 [Debian 档案库](http://deb.debian.org/debian/){.ulink}。

APT 系统定义它的数据源作为[**源列表**]{.strong}，在 `sources.list`{.literal}(5) 里面描述。

对于使用典型的 HTTP 访问的 `trixie`{.literal}系统，[**源列表**]{.strong} 是在 \"`/etc/apt/sources.list.d/debian.sources`{.literal}\" 中由 现代的如下的 deb822 式样提供 ：

``` screen
Suites: trixie
Components: main non-free-firmware contrib non-free

Types: deb deb-src
URIs: http://security.debian.org/debian-security/
Suites: trixie-security
Components: main non-free-firmware contrib non-free
```

deb822 式样的[**源文件**]{.strong} 的关键点如下。

::: itemizedlist
-   它的定义文件在\"`/etc/apt/sources.list.d/*.sources`{.literal}\"文件里。

-   由空格隔开的每个多行块，定义了 APT 系统的数据源。

-   \"`Types:`{.literal}\" 章节定义列表类型，即\"`deb`{.literal}\" 和 \"`deb-src`{.literal}\"。

-   \"`URIs:`{.literal}\"章节定义 Debian 档案库 URI 的根地址。

-   \"`Suites:`{.literal}\"章节定义了发行版名称列表，名称可以使用套件名或代号。

-   \"`Components:`{.literal}\" 章节定义 Debian 档案库中有效档案库名称列表。
:::

如果只是用 `aptitude`{.literal}，它不访问源代码相关的元数据，"`deb-src`{.literal}"定义可以安全地省略。这可以加速档案库元数据的更新。

URL 可以是 \"`https://`{.literal}\", \"`http://`{.literal}\", \"`ftp://`{.literal}\", \"`file://`{.literal}\",......

\"`#`{.literal}\" 开头的行是注释，被忽略。

Debian 开发者（DD）每次上传软件包到 `unstable`{.literal} 档案库（通过 [incoming](https://incoming.debian.org/){.ulink} 处理），都必须确保上传的软件包与最新的 `unstable`{.literal} 档案库中的最新软件包兼容。

在 Debian 档案库维护脚本将软件包从 `unstable`{.literal} 档案库移动到 `testing`{.literal} 档案库前，档案库维护脚本不仅检查时间（约2-10天）和软件包的 RC bug 报告的状态，还尝试确保它们可以和最新的 `testing`{.literal} 档案库中的软件兼容。这个过程使得`testing`{.literal} 档案库非常正确可用。

通过由发布团队领导的逐步冻结档案库的过程，并进行一些手动干预，使 `testing`{.literal} 档案库完全一致，无缺陷。然后，将旧的 `testing`{.literal} 档案库的代码名称分配给新的 `stable`{.literal} 档案库，并为新的 `testing`{.literal} 档案库创建新的代码名称。新的 `testing`{.literal} 档案库最初的内容和新发布的 `stable`{.literal} 档案库的内容完全相同。

`unstable`{.literal} 和 `testing`{.literal} 档案库都可能会遭受由以下几个因素导致的临时的小故障。

::: itemizedlist
-   损坏的软件包被上传到档案库（多见于 `unstable`{.literal} ）

-   延迟接受新的软件包到档案库（多见于 `unstable`{.literal} )

-   档案库时间同步问题（ `testing`{.literal} 和 `unstable`{.literal} ）

-   手动干预档案库，例如移除软件包（多见于 `testing`{.literal} ）等。
:::

因此，如果你决定使用这些档案库，你应该能够修复或忍受这些类型的小故障。

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    在新的 `stable`{.literal} 版本发布后的几个月，大多数桌面用户应该使用带有安全更新的 `stable`{.literal} 档案库，即使他们通常使用 `unstable`{.literal} 或 `testing`{.literal} 档案库。在这个过渡期中，`unstable`{.literal} 和 `testing`{.literal} 档案库不适合大多数人。你使用 `unstable`{.literal} 档案库的系统是很难保持良好的工作状态的，因为它会遭受核心软件包的大量升级狂潮。 `testing`{.literal} 档案库不大有用，因为它包含有和没有安全支持的 `stable`{.literal} 档案库相同的内容（[Debian testing 安全公告 2008-12](https://lists.debian.org/debian-testing-security-announce/2008/12/msg00019.html){.ulink}）。一个月左右的时间后， 如果你仔细点的话， `unstable`{.literal} 或 `testing`{.literal} 档案库或许可以使用。

### []{#_debian_is_100_free_software}2.1.6. Debian 是100% 的自由软件 {#debian-是100-的自由软件 .title}

</div>

</div>
:::::

Debian 是100%的自由软件，因为：

::: itemizedlist
-   Debian 默认只安装自由软件，这尊重了用户的自由。

-   Debian 在 `main`{.literal} 中只提供自由软件。

-   Debian 建议只运行来自 `main`{.literal} 的自由软件。

-   在 `main`{.literal} 中的软件包，没有依赖或推荐在 `non-free`{.literal} 或 `non-free-firmware`{.literal} 或 `contrib`{.literal} 中的软件包。
:::

有人想知道下列的两个事实是否互相矛盾。

::: itemizedlist
-   " Debian 将始终是 100% 的自由软件"。（ [Debian 社群契约](https://www.debian.org/social_contract){.ulink}中的第一条）

-   Debian 服务器上有一些 `non-free-firmware`{.literal}、`non-free`{.literal} 和 `contrib`{.literal} 软件包。
:::

因为下列原因，这并不矛盾。

::: itemizedlist
-   Debian 系统具有100%的自由，并且它的软件包位于 Debian 服务器的 `main`{.literal} 区域。

-   Debian 系统之外的软件包位于 Debian 服务器的 `non-free`{.literal}、 `non-free-firmware`{.literal}和 `contrib`{.literal} 区域。
:::

在 [Debian 社群契约](https://www.debian.org/social_contract){.ulink}的第4条和第5条对这进行了明确的解释：

::: itemizedlist
-   我们将优先考虑我们的用户及自由软件

    ::: itemizedlist
    -   我们由我们的用户及自由软件社群的需要所导向。我们将优先考虑他们的利益。我们将在多种计算环境中支持我们的用户的操作需要。我们不反对在 Debian 系统上使用非自由软件，我们也不会尝试向创建和使用这部分软件的用户索取费用。我们允许他人，在没有我们的资金的参与下，制造包括 Debian 以及商业软件的增值套件。为了达成这些目标，我们将提供集成的、高质量的、100%自由的软件，而不附加任何可能阻止在这些方面使用的法律限制。
    :::

-   哪些作品不符合我们的自由软件规范

    ::: itemizedlist
    -   我们知道，某些我们的用户需要使用不符合 Debian 自由软件指导方针的作品。 我们为这些作品，在我们的档案库中留出了\"`non-free`{.literal}\"、\"`non-free-firmware`{.literal}\"和\"`contrib`{.literal}\"目录。在这些目录下的软件包，并不属于 Debian 系统 尽管它们已被配置成可以在 Debian 下使用。我们鼓励光盘制造商阅读这些目录下的软件的许可证，以判断他们是否可以在光盘中发行这些软件 。所以，尽管非自由软件并非 Debian 系统的一部分，我们仍支持它们的使用，并且我们为非自由软件提供了公共资源 (诸如我们的缺陷跟踪系统以及邮件列表)。Debian 官方媒介可以包括固件，固件不是 Debian 系统的一部分，这是一个例外，能够让 Debian 用于需要这些固件的硬件上。
    :::
:::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 在目前的 [Debian 社群契约（Debian Social Contract）](https://www.debian.org/social_contract){.ulink} 1.2 版本第 5 条条款的实际文本和上面的文本有稍微不同。在不改变 Debian 社群契约实际内容下，这个文字调整让本用户文档在逻辑上保持一致。
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

用户应该了解使用 `non-free`{.literal} 、`non-free-firmware`{.literal}和 `contrib`{.literal} 中的软件包所需要冒的风险：

::: itemizedlist
-   使用类似的软件包会失去自由

-   失去 Debian 对软件包的支持（这些软件包无法访问源代码，Debian 不能进行完全的支持。）

-   污染你100%自由的 Debian 系统
:::

[Debian 自由软件指导方针](https://www.debian.org/social_contract#guidelines){.ulink}为 [Debian](https://www.debian.org){.ulink} 设立了自由软件标准。Debian 对软件包中的软件做了最广泛的解释，包含文档、固件、图标和图形数据。这使得 Debian 的自由软件标准非常严格。

典型的 `non-free`{.literal}、 `non-free-firmware`{.literal}和 `contrib`{.literal} 软件包包含了下列类型的自由分发的软件包：

::: itemizedlist
-   在[GNU Free Documentation License](https://zh.wikipedia.org/wiki/GNU_Free_Documentation_License){.ulink}下的文档包 ，包含不变的部分 ，比如GCC 和 Make的。 （大多数都可以在`non-free/doc`{.literal}找到 .)

-   包含没有源代码的二进制数据的固件软件包，例如在 [第 9.10.5 节 "硬件驱动和固件"](ch09.zh-cn.html#_hardware_drivers_and_firmware "9.10.5. 硬件驱动和固件"){.xref} 中作为 `non-free-firmware`{.literal} 列出的软件包。（多见于 `non-free-firmware/kernel`{.literal} 部分。）

-   游戏和字体软件包，对商业使用和/或内容修改进行了限制。
:::

请注意，`non-free`{.literal}、`non-free-firmware`{.literal} 和 `contrib`{.literal} 软件包的数量少于 `main`{.literal} 软件包的2%。允许访问 `non-free`{.literal}、`non-free-firmware`{.literal} 和 `contrib`{.literal} 并不会模糊软件包的来源。使用 `aptitude`{.literal}(8) 的全屏交互式界面可以提供完全的可见性和完全的控制，可以让你决定安装来自某个部分的软件包，来使你的系统保持自由。

### []{#_package_dependencies}2.1.7. 软件包依赖关系 {#软件包依赖关系 .title}

</div>

</div>
:::::

Debian 系统通过其控制文件字段中的版本化二进制依赖声明机制来提供一致的二进制软件包集合。下面有一些它们的简单定义。

::: itemizedlist
-   "依赖"

    ::: itemizedlist
    -   绝对的依赖，所有在这里列出的软件包都必须同时或提前安装。
    :::

-   \"预依赖\"

    ::: itemizedlist
    -   类似于 Depends，但列出的软件包必须提前完成安装。
    :::

-   \"推荐\"

    ::: itemizedlist
    -   这里表示一个强，但不是绝对的依赖关系 。大多数用户不会想要这个包，除非在这里列出的所有包都已经安装。
    :::

-   \"建议\"

    ::: itemizedlist
    -   较弱的依赖。这个软件包的大多数用户可能会从安装所列的软件包中受益，但没有它们也可以有适当的功能。
    :::

-   \"增强\"

    ::: itemizedlist
    -   这里表明一个像建议的弱依赖关系，不装也没关系。
    :::

-   \"破损\"

    ::: itemizedlist
    -   表明一个软件包不兼容一些版本规范。一般的解决方法就是升级列出的所有软件包。
    :::

-   \"冲突\"

    ::: itemizedlist
    -   这表明了绝对的不兼容。为了安装这个软件包必须移除所有列出的软件包。
    :::

-   \"替代\"

    ::: itemizedlist
    -   这表明这个文件安装的文件会替代所列的软件包的文件。
    :::

-   \"提供\"

    ::: itemizedlist
    -   表明这个软件包会提供所列的软件包所有的文件和功能。
    :::
:::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 请注意，同时将 "Provides" 、"Conflicts" 和 "Replaces" 定义到一个虚拟的软件包是一个明智的配置。这确保了在任何一个时间只能安装一个提供该虚拟包的真正软件包。
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

包含源代码依赖关系的官方定义位于 [the Policy Manual: Chapter 7 - Declaring relationships between packages](https://www.debian.org/doc/debian-policy/ch-relationships){.ulink}。

### []{#_the_event_flow_of_the_package_management}2.1.8. 包管理的事件流 {#包管理的事件流 .title}

</div>

</div>
:::::

这是 APT 提供的软件包管理的简单事件流摘要。

::: itemizedlist
-   [**更新**]{.strong}（"`apt update`{.literal}"、"`aptitude update`{.literal}" 或 "`apt-get update`{.literal}"）：

    ::: orderedlist
    1.  从远程档案库获取档案库元数据

    2.  重建和更新 APT 使用的本地元数据
    :::

-   [**升级**]{.strong}（"`apt upgrade`{.literal}" 和 "`apt full-upgrade`{.literal}"，或 "`aptitude safe-upgrade`{.literal}" 和 "`aptitude full-upgrade`{.literal}"，或 "`apt-get upgrade`{.literal}" 和 "`apt-get dist-upgrade`{.literal}"）：

    ::: orderedlist
    1.  选择候选版本，它所安装的软件包通常都是最新的可用版本（例外参见 [第 2.7.7 节 "使用 [**apt-pinning**]{.strong} 调整获选版本"](ch02.zh-cn.html#_tweaking_candidate_version "2.7.7. 使用 apt-pinning 调整获选版本"){.xref} ）

    2.  解决软件包依赖关系

    3.  如果候选版本与已安装的版本不同，会从远程档案库获取所选择的二进制软件包

    4.  解包所获取的二进制软件包

    5.  运行 [**preinst**]{.strong} 脚本

    6.  安装二进制文件

    7.  运行 [**postinst**]{.strong} 脚本
    :::

-   [**安装**]{.strong}（"`apt install ...`{.literal}"、"`aptitude install ...`{.literal}\" 或者 \"`apt-get install …`{.literal}" ）：

    ::: orderedlist
    1.  选择命令行中列出的包

    2.  解决软件包依赖关系

    3.  从远程服务器获取已选二进制包

    4.  解包所获取的二进制软件包

    5.  运行 [**preinst**]{.strong} 脚本

    6.  安装二进制文件

    7.  运行 [**postinst**]{.strong} 脚本
    :::

-   [**移除**]{.strong} （\"`apt remove …`{.literal}\", "`aptitude remove …`{.literal}" 或 "`apt-get remove …`{.literal}"）：

    ::: orderedlist
    1.  选择命令行中列出的包

    2.  解决软件包依赖关系

    3.  运行 [**prerm**]{.strong} 脚本

    4.  移除已安装的文件，[**除了**]{.strong}配置文件

    5.  运行 [**postrm**]{.strong} 脚本
    :::

-   [**清除**]{.strong} （\"`apt purge`{.literal}\","`aptitude purge …`{.literal}" 或 "`apt-get purge …`{.literal}"）：

    ::: orderedlist
    1.  选择命令行中列出的包

    2.  解决软件包依赖关系

    3.  运行 [**prerm**]{.strong} 脚本

    4.  移除已安装的文件，[**包含**]{.strong}配置文件

    5.  运行 [**postrm**]{.strong} 脚本
    :::
:::

这里，为了大局，我特意省略了技术细节。

### []{#_how_to_pick_debian_packages}2.1.10. 如何挑选 Debian 软件包 {#如何挑选-debian-软件包 .title}

::: itemizedlist
-   重要性：是 \> 否

-   类型：main \> contrib \> non-free

-   优先级：需要 \> 重要 \> 标准 \> 可选 \> 额外

-   任务：在任务下有软件包的列表信息，例如 "桌面环境"

-   软件包是被与之有依赖关系的软件包所选择的（例如 `gcc`{.literal} 依赖 `gcc-10`{.literal}）

-   流行度：在投票或者安装指数上有着更高的分数

-   更新日志：维护者经常的更新

-   BTS (缺陷跟踪系统):没有 RC 级别的缺陷（没有危险、重大严重的缺陷）

-   BTS (缺陷跟踪系统):有维护者对缺陷报告反馈

-   BTS (缺陷跟踪系统): 有着更多的近期修复的 bug 数目

-   BTS (缺陷跟踪系统):遗留的非严重（non-wishlist）缺陷数量较少
:::

Debian 是一个使用分布式开发模式的志愿项目，它的档案库包含了许多不同关注点和不同质量的软件包。你必须做出自己的选择。

### []{#_how_to_cope_with_conflicting_requirements}2.1.11. 怎样和不一致的要求协作 {#怎样和不一致的要求协作 .title}

</div>

</div>
:::::

无论你决定使用哪个 Debian 系统套件，你仍然希望运行在那个套件里不存在的程序版本。即使你在其它 Debian 套件里面，或者在其它非 Debian 的资源里面，找到这个程序的二进制软件包，它们的要求可能和你目前的 Debian 系统不一致。

在 [第 2.7.7 节 "使用 [**apt-pinning**]{.strong} 调整获选版本"](ch02.zh-cn.html#_tweaking_candidate_version "2.7.7. 使用 apt-pinning 调整获选版本"){.xref} 里描述的 [**apt-pinning**]{.strong} 等技术，虽然你能够用它来调整软件包管理系统来安装这类不同步的二进制软件包，但这样的调整方法只有有限的使用场景，因为它们可能破坏那些程序和你的系统。

在单独安装这类不同步的软件包之前，你需要查找所有存在的和你目前 Debian 系统兼容的安全技术替代方案。

::: itemizedlist
-   使用相应的沙盒化的上游二进制软件包来安装这样的程序（参见 [第 7.7 节 "沙盒"](ch07.zh-cn.html#_sandbox "7.7. 沙盒"){.xref}）。

    ::: itemizedlist
    -   许多常见的 GUI（图形用户界面）程序，比如 LibreOffice 和 GNOME 应用，会有 [Flatpak](https://zh.wikipedia.org/wiki/Flatpak){.ulink}、[Snap](https://zh.wikipedia.org/wiki/Snap_(software)){.ulink} 或 [AppImage](https://zh.wikipedia.org/wiki/AppImage){.ulink} 软件包存在。
    :::

-   建立一个 chroot 或类似的环境来在里面运行这样的程序（参见[第 9.11 节 "虚拟化系统"](ch09.zh-cn.html#_virtualized_system "9.11. 虚拟化系统"){.xref}）。

    ::: itemizedlist
    -   CLI 命令能够在和它兼容的 chroot 下轻松执行（参见 [第 9.11.4 节 "Chroot 系统"](ch09.zh-cn.html#_chroot_system "9.11.4. Chroot 系统"){.xref}）。

    -   不重启机器，能够轻松的尝试多个完整的桌面环境（参见 [第 9.11.5 节 "多桌面系统"](ch09.zh-cn.html#_multiple_desktop_systems "9.11.5. 多桌面系统"){.xref}）。
    :::

-   自己构建需要的二进制软件包版本，和你目前 Debian 系统兼容。

    ::: itemizedlist
    -   这是一个 [不轻松的任务](https://www.debian.org/doc/manuals/debmake-doc/){.ulink} (参见 [第 2.7.13 节 "移植一个软件包到 stable 系统"](ch02.zh-cn.html#_porting_a_package_to_the_stable_system "2.7.13. 移植一个软件包到 stable 系统"){.xref})。
    :::

## []{#_basic_package_management_operations}2.2. 基础软件包管理操作 {#基础软件包管理操作 .title}

</div>

</div>
:::::

在 Debian 系统中有许多基于 APT 的软件包管理工具可以在 Debian 系统上进行基于仓库的软件包管理操作。在这里，我们将介绍 3 种基本的软件包管理工具：`apt`{.literal},`apt-get`{.literal} / `apt-cache`{.literal} 和 `aptitude`{.literal} 。

对于涉及软件包安装或更新软件包元数据的软件包管理操作，你必须有 root 权限。

::::::::::: section
::::: titlepage
<div>

<div>

### []{#_literal_apt_literal_vs_literal_apt_get_literal_literal_apt_cache_literal_vs_literal_aptitude_literal}2.2.1. `apt`{.literal} vs. `apt-get`{.literal} / `apt-cache`{.literal} vs. `aptitude`{.literal} {#apt-vs.-apt-get-apt-cache-vs.-aptitude .title}

</div>

</div>
:::::

尽管 `aptitude`{.literal} 是作者主要使用的一个非常好的可互动工具，但你应该知道下列警示：

::: itemizedlist
-   不建议在新版本发布后在 `stable`{.literal} Debian 系统上使用 `aptitude`{.literal} 命令来进行跨版本的系统升级。

    ::: itemizedlist
    -   建议使用\"`apt full-upgrade`{.literal}\"或"`apt-get dist-upgrade`{.literal}" 来进行这个操作。参见 [Bug #411280](https://bugs.debian.org/411280){.ulink}。
    :::

-   `aptitude`{.literal}命令有时候会为了`testing`{.literal} 或 `unstable`{.literal} Debian 系统升级清除大量软件包。

    ::: itemizedlist
    -   这个情况吓坏了许多的系统管理员。请不要惊慌。

    -   这似乎大多数是由元软件包的依赖或推荐的软件包版本偏差造成的，例如 `gnome-core`{.literal}。

    -   要解决这个问题，可以在 `aptitude`{.literal} 命令菜单中选择 "取消待执行的动作" ，退出 `aptitude`{.literal}，并使用 "`apt full-upgrade`{.literal}"。
    :::
:::

`apt-get`{.literal} 和 `apt-cache`{.literal} 是最[**基础**]{.strong} 的基于 APT 的软件包管理工具。

::: itemizedlist
-   `apt-get`{.literal} 和 `apt-cache`{.literal} 只提供命令行用户界面。

-   `apt-get`{.literal} 是进行跨版本的[**主系统升级**]{.strong}等操作的最合适工具。

-   `apt-get`{.literal} 提供了一个[**强大**]{.strong}的软件包依赖解析器。

-   `apt-get`{.literal} 对硬件资源的要求不高。它消耗更少的内存并且运行速度更快。

-   `apt-cache`{.literal} 提供了一个 [**标准的**]{.strong}正则表达式来搜索软件包名称和描述。

-   `apt-get`{.literal} 和 `apt-cache`{.literal} 可以使用 `/etc/apt/preferences`{.literal} 来管理软件包的多个版本，但这非常繁琐。
:::

`apt`{.literal} 命令是一个用于软件包管理的高级命令行界面。它基本上是 `apt-get`{.literal}、`apt-cache`{.literal} 和类似命令的一个封装，被设计为针对终端用户交互的界面，它默认启用了某些适合交互式使用的选项。

::: itemizedlist
-   `apt`{.literal} 工具在用户使用 `apt install`{.literal} 安装软件包时提供了一个友好的进度条。

-   在成功安装下载的软件包后，`apt`{.literal} 将默认[**删除**]{.strong}缓存的 `.deb`{.literal} 软件包。
:::

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                建议用户使用新的 `apt`{.literal}(8) 命令用于 [**交互式**]{.strong}的使用场景，而在 shell 脚本中使用 `apt-get`{.literal}(8) 和`apt-cache`{.literal}(8) 命令。
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

`aptitude`{.literal} 命令是最[**通用的**]{.strong}基于 APT 的软件包管理工具。

::: itemizedlist
-   `aptitude`{.literal} 提供了一个全屏的交互式文本用户界面。

-   `aptitude`{.literal} 同样也提供了一个命令用户界面。

-   `aptitude`{.literal} 是用于[**日常软件包管理**]{.strong}（例如检查已安装的软件包和搜索可用的软件包）的最合适工具。

-   `aptitude`{.literal} 对硬件资源的要求更高。它消耗更多的内存并且运行速度更慢。

-   `aptitude`{.literal} 提供一个[**增强的**]{.strong}正则表达式来搜索所有的软件包元数据。

-   `aptitude`{.literal} 可以管理软件包的多个版本，并且不使用 `/etc/apt/preferences`{.literal}，这会十分直观。

::: itemizedlist
-   正则表达式使用的是 [**ERE**]{.strong}，就跟 `egrep`{.literal}(1)、`awk`{.literal}(1) 和 `perl`{.literal}(1) 这些典型的类 Unix 文本工具中所使用的 "`^`{.literal}"、"`.*`{.literal}"、"`$`{.literal}" 等是相同的。

-   依赖关系 [*type*]{.emphasis} 是一种特定的软件包相互关系（depends、predepends、recommends、suggests、conflicts、replaces、provides）。

-   默认的 [*type*]{.emphasis} 依赖关系是 "depends"。

  ----------------------------- -----------------------------------------------------------------------------------------

::: itemizedlist
-   \"`~P`{.literal}[*`term`{.literal}*]{.emphasis}\" == \"`~Dprovides:`{.literal}[*`term`{.literal}*]{.emphasis}\"

-   \"`~C`{.literal}[*`term`{.literal}*]{.emphasis}\" == \"`~Dconflicts:`{.literal}[*`term`{.literal}*]{.emphasis}\"

-   \"`…~W term`{.literal}\" == \"`(…|term)`{.literal}\"
:::

### []{#_tidying_auto_manual_install_status}2.3.5. 调整自动/手动安装状态 {#调整自动手动安装状态 .title}

</div>

</div>
:::::

下面是调整软件包的自动/手动安装状态的方法（在使用非 aptitude 软件包管理器之后）。

::: orderedlist
1.  用 root 以交互模式运行 `aptitude`{.literal}。

2.  用 \"`u`{.literal}\" 命令更新可用的软件包列表，\"`U`{.literal}\" 命令标记所有可升级的软件包以执行升级，\"`f`{.literal}\" 命令清除新软件包列表，\"`g`{.literal}\" 命令执行所有可升级的软件包以执行升级。

3.  按下 "`l`{.literal}"，并输入 "`~i(~R~i|~Rrecommends:~i)`{.literal}" 来限制软件包的显示，按下 "`M`{.literal}" 将 "`已安装软件包`{.literal}" 的状态改为自动安装。

4.  按下 "`l`{.literal}"，并输入 "`~prequired|~pimportant|~pstandard|~E`{.literal}" 来限制软件包的显示，按下 "`m`{.literal}" 将 "`已安装软件包`{.literal}" 的状态改为手动安装。

5.  按下 "`l`{.literal}"，并输入 "`~i!~M`{.literal}" 来限制软件包的显示，在 "`已安装软件包`{.literal}" 上按下 "`[`{.literal}" 来陈列无用的软件包，按下 "`-`{.literal}" 将它们移除。

6.  按下 "`l`{.literal}"，并输入 "`~i`{.literal}" 来限制软件包的显示，之后在 "`软件集`{.literal}" 上按下 "`m`{.literal}" 将那些软件包标记为手动安装。

7.  退出 `aptitude`{.literal}。

8.  用 root 用户执行 \"`apt-get -s autoremove|less`{.literal}\" 命令，来查看有那些软件包是不再需要的。

9.  在交互模式下重启 `aptitude`{.literal} 程序，用 "`m`{.literal}" 命令标记所需要的软件包。

10. 用 root 用户重新执行 \"`apt-get -s autoremove|less`{.literal}\" 这个命令来复查移除的包中是不是只含有自己所希望移除的软件包。

11. 用 root 用户执行 \"`apt-get autoremove|less`{.literal}\" 命令来自动移除不再需要的软件包。
:::

在你所需要执行的 \"`Tasks`{.literal}\" 上，运行 \"`m`{.literal}\" 命令是一个可选的操作，目的就是为了防止大量软件包被卸载的情况出现。

### []{#_system_wide_upgrade}2.3.6. 全面的系统升级 {#全面的系统升级 .title}

</div>

</div>
:::::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 当你迁移到新的发行版的时候，虽然正如下面所描述的那样，Debian 是可升级的，但是你还是应该考虑纯净的安装新的系统。这给了你机会去移除废弃的软件包同时还可以接触到最新软件包的完美集合体。当然，在做迁移之前，你也应该对你的系统做完整的备份，并把它移到安全的地方去（查看[第 10.2 节 "备份和恢复"](ch10.zh-cn.html#_backup_and_recovery "10.2. 备份和恢复"){.xref})。"我"也建议用不同的分区做另外一个启动项，来实现平稳的升级。
  ------------------------------ -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

你可以通过改变 [**源列表**]{.strong} 的内容使之指向新的发行版所在地址的方法来进行系统的全面升级，然后运行 \"`apt update; apt dist-upgrade`{.literal}\" 命令。

在 `trixie`{.literal}-作为-`stable`{.literal} 发布循环中, 从 `stable`{.literal} 升级到 `testing`{.literal} 或者 `unstable`{.literal}，你应该用 \"`forky`{.literal}\" 或者 \"`sid`{.literal}\" 替换[**源列表**]{.strong} 文件里的 \"`trixie`{.literal}\"示例，参考 [第 2.1.5 节 "Debian 档案库基础"](ch02.zh-cn.html#_debian_archive_basics "2.1.5. Debian 档案库基础"){.xref}。

事实上，由于一些软件包版本变迁的问题，你可能会遇到一些困难，主要是由于软件包的依赖问题。升级之后的差异越大，你越有可能遇到麻烦。在新版本发行后，系统从旧的 `stable`{.literal} 过渡到新的 ` stable`{.literal}，你可以查看 [Release Notes](https://www.debian.org/releases/stable/releasenotes){.ulink} 然后按照里面的步骤去做，来尽可能的减少麻烦。

在它正式发布之前，你决定要从先前的 ` stable`{.literal} 迁移到将要发布的 ` testing`{.literal}，这里没有 [Release Notes](https://www.debian.org/releases/stable/releasenotes){.ulink} 可以帮到你。 在前一个 `stable`{.literal} 发布以后，` stable`{.literal} 发行版跟将要发布的` testing`{.literal} 发行版之间的差异可能变得相当大同时也使得升级系统变得更加的复杂。

在全面升级系统的时候，你应该谨慎的操作，同时你也应该从邮件列表中获取最新的资料然后根据你的常识作出正确的判断。

::: orderedlist
1.  查看先前的"发行说明"。

2.  备份整个系统(尤其是数据和配置信息）。

3.  当 bootloader 坏了的时候，手边应该有可以引导电脑启动的存储介质。

4.  事先通知系统上的用户。

5.  用` script`{.literal}(1) 记录升级的过程。

6.  用 "unmarkauto" 命令来保留你想要的软件包，例如 \"`aptitude unmarkauto vim`{.literal}\" 这个命令是用来防止移除 vim 这个软件的。

7.  为了减少软件包之间可能会发生的冲突，应该尽量减少要安装的软件包的数目，例如，移除桌面环境这个软件包。

8.  移除 \"`/etc/apt/preferences`{.literal}\" 文件（禁用[**apt-pinning**]{.strong}）。

9.  试着一步步的升级： `oldstable`{.literal} → `stable`{.literal} → `testing`{.literal} → `unstable`{.literal}.

10. 升级 [**源列表**]{.strong} 文件，使其指向新的档案库然后运行 \"`aptitude update`{.literal}\" 命令。

11. 可选的安装选项，首先是新的 [ **core packages**]{.strong}，例如 "`aptitude install perl`{.literal}\"。

12. 运行 \"`apt-get -s dist-upgrade`{.literal}\" 命令来评估升级造成的影响。

13. 最后运行 \"`apt-get dist-upgrade`{.literal}\" 命令。
:::

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- -----------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    在 ` stable`{.literal} 版本升级的时候，跳过主要的 Debian 发行版是不明智的。
  --------------------------------- -----------------------------------------------------------------------------
:::

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    在先前的"发行手册"里， GCC、 Linux 内核、 initrd-tools、 Glibc、 Perl、 APT 工具链等等，有一些关于系统全面升级的重要注意事项。
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------
:::

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- -----------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    "发行说明"不一定覆盖所有可能情况。如果你修改了底层配置，下一次升级可能会因为[\"\... 升级后的段错误 \...\"](https://bugs.debian.org/1112535){.ulink}而严重失败。
  --------------------------------- -----------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

关于 ` unstable`{.literal} 版本的日常升级，查看 [第 2.4.3 节 "预防软件包故障"](ch02.zh-cn.html#_safeguarding_for_package_problems "2.4.3. 预防软件包故障"){.xref}。

请注意以下几点。

::: itemizedlist
-   所有的系统配置和安装命令都需要以 root 运行。

-   不同于使用正则表达式的 `aptitude`{.literal} （参见 [第 1.6.2 节 "正则表达式"](ch01.zh-cn.html#_regular_expressions "1.6.2. 正则表达式"){.xref}），其它的软件包管理命令使用类似于 shell glob 的通配符（参见 [第 1.5.6 节 "Shell 通配符"](ch01.zh-cn.html#_shell_glob "1.5.6. Shell 通配符"){.xref}）。

-   `apt-file`{.literal}(1) 由 `apt-file`{.literal} 软件包提供，并且需要先运行 "`apt-file update`{.literal}"。

-   `configure-debian`{.literal}(8) 由 `configure-debian`{.literal} 软件包提供，它运行 `dpkg-reconfigure`{.literal}(8) 作为后端。

-   `dpkg-reconfigure`{.literal}(8) 使用 `debconf`{.literal}(1) 作为后端来运行软件包脚本。

-   \"`apt-get build-dep`{.literal}\" 、\"`apt-get source`{.literal}\" 和 \"`apt-cache showsrc`{.literal}\" 命令需要 [**源列表**]{.strong}中存在 \"`deb-src`{.literal}\" 条目。

-   `dget`{.literal}(1) 、`debuild`{.literal}(1) 和 `debi`{.literal}(1) 需要 `devscripts`{.literal} 软件包。

-   参见 [第 2.7.13 节 "移植一个软件包到 stable 系统"](ch02.zh-cn.html#_porting_a_package_to_the_stable_system "2.7.13. 移植一个软件包到 stable 系统"){.xref} 里使用 \"`apt-get source`{.literal}\" 的打包（重打包）过程。

### []{#_safeguarding_for_package_problems}2.4.3. 预防软件包故障 {#预防软件包故障 .title}

许多用户更想使用 Debian 系统的 [**testing**]{.strong} （或 [**unstable**]{.strong}）版本，因为它有新的功能和软件包。但这会使得系统更容易遇到严重的软件包问题。

### []{#_searching_on_the_package_meta_data}2.4.4. 搜索软件包元数据 {#搜索软件包元数据 .title}

</div>

</div>
:::::

尽管近来浏览 Debian 网站 [https://packages.debian.org/](https://packages.debian.org/){.ulink} 是搜索软件包元数据更加简单的方法，但我们依旧来看看更传统的方法。

`grep-dctrl`{.literal}(1) 、`grep-status`{.literal}(1) 和 `grep-available`{.literal}(1) 命令被用来搜索具有 Debian 软件包控制文件格式的任何文件。

如果你想递归查找软件包依赖，你应该使用 `apt-rdepends`{.literal}(8) 。

### []{#_top_level_release_file_and_authenticity}2.5.2. 顶层"Release"文件及真实性 {#顶层release文件及真实性 .title}

</div>

</div>

  ----------------------------- ------------------------------------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                顶层"`Release`{.literal}"文件用于签署 [**secure APT**]{.strong} 系统下的归档文件。

每个 Debian 档案库的网址都有一个这样的 "`Release`{.literal}" 文件，例如 "`http://deb.debian.org/debian/dists/unstable/Release`{.literal}"，内容如下。

``` screen
Origin: Debian
Label: Debian
Suite: unstable
Codename: sid
Date: Sat, 14 May 2011 08:20:50 UTC
Valid-Until: Sat, 21 May 2011 08:20:50 UTC
Architectures: alpha amd64 armel hppa hurd-i386 i386 ia64 kfreebsd-amd64 kfreebsd-i386 mips mipsel powerpc s390 sparc
Components: main contrib non-free
Description: Debian x.y Unstable - Not Released
MD5Sum:
 bdc8fa4b3f5e4a715dd0d56d176fc789 18876880 Contents-alpha.gz
 9469a03c94b85e010d116aeeab9614c0 19441880 Contents-amd64.gz
 3d68e206d7faa3aded660dc0996054fe 19203165 Contents-armel.gz
...
```

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 在 [第 2.1.5 节 "Debian 档案库基础"](ch02.zh-cn.html#_debian_archive_basics "2.1.5. Debian 档案库基础"){.xref} 里，你能够发现我使用\"suite\" 和 \"codename\"的逻辑。"发行版"被用来同时谈及\"suite\" 和 \"codename\".所有由档案库提供的归档 \"area\" 名，会被列在 \"Components\" 下.
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

顶层文件 \"`Release`{.literal}\" 的完整性，是由叫 [secure apt](https://wiki.debian.org/SecureApt){.ulink} 的加密架构来验证，在 `apt-secure`{.literal}(8)中进行描述。

::: itemizedlist
-   加密签名文件 \"`Release.gpg`{.literal}\" 是由顶层授权文件 \"`Release`{.literal}\" 和加密的 Debian 档案库公钥创建。

-   公开的 Debian 档案库公钥能够通过安装 `debian-archive-keyring`{.literal} 软件包来安装到本地。

-   [**secure APT**]{.strong} 系统自动验证下载的顶层文件 \"`Release`{.literal}\" 的完整性。加密验证过程用到了\"`Release.gpg`{.literal}\"文件和本地安装的 Debian 档案库公钥.

-   所有 \"`Packages`{.literal}\" 和 \"`Sources`{.literal}\" 文件的完整性是由在顶层 \"`Release`{.literal}\" 文件里的 MD5sum 值来验证。所有软件包文件的完整性由 \"`Packages`{.literal}\" 和 \"`Sources`{.literal}\" 文件里的 MD5sum 值来验证.参见 `debsums`{.literal}(1) 和 [第 2.4.2 节 "验证安装的软件包文件"](ch02.zh-cn.html#_verification_of_installed_package_files "2.4.2. 验证安装的软件包文件"){.xref}.

-   因加密签名验证比计算 MD5sum 值消耗更多的 CPU，使用 MD5sum 值来验证每一个软件包，使用加密签名来验证顶层的 \"`Release`{.literal}\" 文件，这种方式提供 [较好安全性的同时，也有比较好的性能](https://www.infodrom.org/~joey/Writing/Linux-Journal/secure-apt/){.ulink} (参见 [第 10.3 节 "数据安全基础"](ch10.zh-cn.html#_data_security_infrastructure "10.3. 数据安全基础"){.xref}).
:::

如果 [**源列表**]{.strong}条目特别指定了 \"`signed-by`{.literal}\" 选项，它下载的顶层\"`Release`{.literal}\"文件使用这个指定的公钥来验证。这在当[**源列表**]{.strong}包含有非 Debian 档案库时有用。

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}
  ----------------------------- ---------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                不赞成使用 `apt-key`{.literal}(8) 命令来管理 APT 密钥。
  ----------------------------- ---------------------------------------------------------
:::

当然，你能够使用 `gpg`{.literal} 手工验证\"`Release`{.literal}\" 的完整性，使用 \"`Release.gpg`{.literal}\" 文件和在 [ftp-master.debian.org](https://ftp-master.debian.org){.ulink} 上公布的 Debian 档案库公钥。

### []{#_fetching_of_the_meta_data_for_the_package}2.5.4. 获取用于软件包的元数据 {#获取用于软件包的元数据 .title}

</div>

</div>
:::::

当使用 APT 工具时，如 `aptitude`{.literal}, `apt-get`{.literal}, `synaptic`{.literal}, `apt-file`{.literal}, `auto-apt`{.literal}，我们需要更新包含 Debian 档案库信息元数据的本地拷贝。这些本地拷贝的文件名称，和在 [**源列表**]{.strong}文件里面的 `distribution`{.literal}, `area`{.literal}, `architecture`{.literal} 相应名称一致。 (参见 [第 2.1.5 节 "Debian 档案库基础"](ch02.zh-cn.html#_debian_archive_basics "2.1.5. Debian 档案库基础"){.xref}).

::: itemizedlist
-   \"`/var/lib/apt/lists/deb.debian.org_debian_dists_`{.literal}[*`distribution`{.literal}*]{.emphasis}`_Release`{.literal}\"

-   \"`/var/lib/apt/lists/deb.debian.org_debian_dists_`{.literal}[*`distribution`{.literal}*]{.emphasis}`_Release.gpg`{.literal}\"

-   \"`/var/lib/apt/lists/deb.debian.org_debian_dists_`{.literal}[*`distribution`{.literal}*]{.emphasis}`_`{.literal}[*`area`{.literal}*]{.emphasis}`_binary-`{.literal}[*`architecture`{.literal}*]{.emphasis}`_Packages`{.literal}\"

-   \"`/var/lib/apt/lists/deb.debian.org_debian_dists_`{.literal}[*`distribution`{.literal}*]{.emphasis}`_`{.literal}[*`area`{.literal}*]{.emphasis}`_source_Sources`{.literal}\"

-   \"`/var/cache/apt/apt-file/deb.debian.org_debian_dists_`{.literal}[*`distribution`{.literal}*]{.emphasis}`_Contents-`{.literal}[*`architecture`{.literal}*]{.emphasis}`.gz`{.literal}\" (`apt-file`{.literal})
:::

前 4 种类型的文件是所有相关的 APT 命令共享的，并且可以通过 "`apt-get update`{.literal}" 或 "`aptitude update`{.literal}" 在命令行中进行更新。如果在[**源列表**]{.strong}中有相应的 "`deb`{.literal}" 行，则 "`软件包`{.literal}" 元数据会进行更新。如果在 [**源列表**]{.strong}中有相应的 "`deb-src`{.literal}" 行，则 "`源代码`{.literal}" 元数据会进行更新。

\"`Packages`{.literal}\" 和 \"`Sources`{.literal}\" 的元数据文件包含有"`Filename:`{.literal}"字段，指向二进制和源代码包文件的位置。目前，这些软件包都统一放在\"`pool/`{.literal}\"目录树下，这样可以改善跨版本发布的传输。

"`软件包`{.literal}"元数据的本地副本可以使用 `aptitude`{.literal} 来进行交互式的搜索。专门的搜索命令 `grep-dctrl`{.literal}(1) 可以搜索"`软件包`{.literal}"和"`源代码`{.literal}"元数据的本地副本。

\"`Contents-`{.literal}[*`architecture`{.literal}*]{.emphasis}\"元数据的本地拷贝，能够被\"`apt-file update`{.literal}\"更新，它的位置和其它 4 个不同。参见 `apt-file`{.literal}(1). (`auto-apt`{.literal} 的 \"`Contents-`{.literal}[*`architecture`{.literal}*]{.emphasis}`.gz`{.literal}\"文件的本地拷贝默认也使用不同的位置。)

### []{#_the_dpkg_command}2.5.9. dpkg 命令 {#dpkg-命令 .title}

</div>

</div>
:::::

`dpkg`{.literal}(1) 是 Debian 软件包管理中最底层的工具。它非常强大，必须小心使用。

当安装名为 "[*`package_name`{.literal}*]{.emphasis}" 的软件包时，`dpkg`{.literal} 会按照下列的顺序处理它。

::: orderedlist
1.  解包 deb 文件（等同于 "`ar -x`{.literal}"）

2.  使用 `debconf`{.literal}(1) 执行 "[*`package_name`{.literal}*]{.emphasis}`.preinst`{.literal}"

3.  将软件包安装到系统中（等同于 "`tar -x`{.literal}"）

4.  使用 `debconf`{.literal}(1) 执行 "[*`package_name`{.literal}*]{.emphasis}`.postinst`{.literal}"
:::

`debconf`{.literal} 系统提供带有 I18N 和 L10N （[第 8 章 *国际化和本地化*](ch08.zh-cn.html "第 8 章 国际化和本地化"){.xref}）支持的标准化用户交互。

### []{#_incompatibility_with_old_user_configuration}2.6.1. 不兼容旧的用户配置 {#不兼容旧的用户配置 .title}

</div>

### []{#_rescue_with_the_dpkg_command}2.6.3. 使用 dpkg 命令进行救援 {#使用-dpkg-命令进行救援 .title}

</div>

</div>
:::::

因 `dpkg`{.literal} 是非常底层的软件包工具，它能够在没有网络连接时运行。

让我们假设 `foo`{.literal} 软件包损坏了，需要被修复。

你可以在软件包缓存目录："`/var/cache/apt/archives/`{.literal}" 中找到旧的 `foo`{.literal} 软件包的无 bug 版本。（如果找不到，你可以从档案库 [https://snapshot.debian.org/](https://snapshot.debian.org/){.ulink} 中下载它，或从具有软件包缓存功能的机器中拷贝它。）

如果你能够启动系统，你可以通过下列命令来安装它。

``` screen
# dpkg -i /path/to/foo_old_version_arch.deb
```

如果由于依赖问题，无法用这种方式安装软件包，并且你真的必须真么做，你可以使用 `dpkg`{.literal} 的 "`--ignore-depends`{.literal}"、"`--force-depends`{.literal}" 和其它选项来无视依赖。如果你这么做了，之后你必须认真努力地修复依赖关系。更多细节参见 `dpkg`{.literal}(8)。

  ------------------------------ --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 如果你的系统严重损坏了，你应该将系统完整备份到一个安全的地方（参见 [第 10.2 节 "备份和恢复"](ch10.zh-cn.html#_backup_and_recovery "10.2. 备份和恢复"){.xref}）并进行一次全新的安装。这是耗时较少且效果较好的办法。
  ------------------------------ --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

### []{#_different_packages_with_overlapped_files}2.6.5. 具有相同文件的不同软件包 {#具有相同文件的不同软件包 .title}

</div>

</div>
:::::

文档级的软件包管理系统，比如说 `aptitude`{.literal}(8) 或 `apt-get`{.literal}(1), 使用软件包依赖，当出现相同文件时，不会尝试去安装软件包。(参见 [第 2.1.7 节 "软件包依赖关系"](ch02.zh-cn.html#_package_dependencies "2.1.7. 软件包依赖关系"){.xref}).

软件包维护者的错误，或者系统管理员配置了不一致的档案库混合源，(参见 [第 2.7.6 节 "不使用 [**apt-pinning**]{.strong} 的混合源档案库软件包"](ch02.zh-cn.html#_packages_from_mixed_source_of_archives "2.7.6. 不使用 apt-pinning 的混合源档案库软件包"){.xref})，都会出现不正确的软件包依赖情况。如果在出现相同文件的情况下，你通过 `aptitude`{.literal}(8) 或 `apt-get`{.literal}(1) 安装软件包，`dpkg`{.literal}(1) 在对软件包解包时，确定会给调用程序返回错误，并不会覆盖已经存在的文件。

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    使用第三方软件包会导致重大的系统风险，因为其通过使用 root 权限运行维护者脚本能够对你的系统做任何事。`dpkg`{.literal}(1) 命令只防止解包时的覆盖行为。
  --------------------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------
:::

可以先通过删除旧的令人讨厌的软件包，[*`old-package`{.literal}*]{.emphasis}，来解决这类错误的安装问题。

``` screen
$ sudo dpkg -P old-package
```

### []{#_fixing_broken_package_script}2.6.6. 修复损坏的软件包脚本 {#修复损坏的软件包脚本 .title}

</div>

</div>
:::::

当软件包脚本中的一个命令由于某些原因返回错误，脚本也将由于错误而退出，软件包管理系统忽略它们的行为，并导致部分安装的软件包。当一个软件包在它的删除脚本中有错误时，该软件包将会成为不可能删除的软件包，处理这些问题，都会变得相当棘手。

-   \"`/var/lib/dpkg/info/`{.literal}[*`package_name`{.literal}*]{.emphasis}`.postrm`{.literal}\"

::: itemizedlist
-   在行首添加 "`#`{.literal}" 可以禁用出错的行

-   在出错行的行尾添加 "`|| true`{.literal}" 可以强制返回成功
:::

然后，使用下列命令来配置所有部分安装的软件包。

``` screen
# dpkg --configure -a
```

### []{#_recovering_package_selection_data}2.6.7. 恢复软件包选择数据 {#恢复软件包选择数据 .title}

</div>

</div>
:::::

如果 "`/var/lib/dpkg/status`{.literal}" 因为某种原因出现错误，Debian 系统会丢失软件包选择数据并受到严重影响。寻找位于 "`/var/lib/dpkg/status-old`{.literal}" 或 "`/var/backups/dpkg.status.*`{.literal}" 中旧的 "`/var/lib/dpkg/status`{.literal}" 文件。

给 "`/var/backups/`{.literal}" 分配一个单独的分区是一个好习惯，因为这个目录包含了许多重要的系统数据。

对于严重的损坏，我建议备份系统后重新安装。即使失去 "`/var/`{.literal}" 中的所有数据，你依旧可以从 "`/usr/share/doc/`{.literal}" 目录恢复一些信息来引导你进行新的安装。

重新安装最小（桌面）系统。

``` screen
# mkdir -p /path/to/old/system
```

将旧系统挂载到 "`/path/to/old/system/`{.literal}"。

``` screen
# cd /path/to/old/system/usr/share/doc
# ls -1 >~/ls1.txt
# cd /usr/share/doc
# ls -1 >>~/ls1.txt
# cd
# sort ls1.txt | uniq | less
```

然后你就可以根据软件包名称来进行安装了。（可能会有一些非软件包名称，例如 "`texmf`{.literal}"。）

### []{#_automatic_download_and_upgrade_of_packages}2.7.3. 自动下载和升级软件包 {#自动下载和升级软件包 .title}

</div>

</div>
:::::

`apt`{.literal} 软件包有自己的 cron 脚本 "`/etc/cron.daily/apt`{.literal}" ，它支持自动下载软件包。可以安装 `unattended-upgrades`{.literal} 软件包来增强这个脚本，使它能够自动升级软件包。可以通过 "`/etc/apt/apt.conf.d/02backup`{.literal}" 和 "`/etc/apt/apt.conf.d/50unattended-upgrades`{.literal}" 中的参数来进行自定义，相关说明位于 "`/usr/share/doc/unattended-upgrades/README`{.literal}" 中。

`unattended-upgrades`{.literal} 软件包主要用于 `stable`{.literal} 系统的安全更新。如果自动升级损坏 `stable`{.literal} 系统的风险小于被入侵者利用已被安全更新修复的安全漏洞，你应该考虑使用自动更新，配置参数如下。

``` screen
APT::Periodic::Update-Package-Lists "1";
APT::Periodic::Download-Upgradeable-Packages "1";
APT::Periodic::Unattended-Upgrade "1";
```

如果你运行的是 `testing`{.literal} 或 `unstable`{.literal} 系统，你应该不会想要使用自动更新，因为它肯定会在某天损坏系统。即使位于这样的 `testing`{.literal} 或`unstable`{.literal} 情况下，你可能依旧想提前下载软件包以节省交互式升级的时间，其配置参数如下。

``` screen
APT::Periodic::Update-Package-Lists "1";
APT::Periodic::Download-Upgradeable-Packages "1";
APT::Periodic::Unattended-Upgrade "0";
```

### []{#_updates_and_backports}2.7.4. 更新和向后移植 {#更新和向后移植 .title}

</div>

</div>
:::::

[stable-updates](https://wiki.debian.org/StableUpdates){.ulink} (\"`trixie`{.literal}-updates\"，在 `trixie`{.literal}-作为-`stable`{.literal} 发布循环) 和 [backports.debian.org](https://backports.debian.org){.ulink} 档案库提供了 `stable`{.literal} 版软件包更新。

为了去使用这些档案库，你需要在 \"`/etc/apt/sources.list`{.literal}\" 文件里写入如下所示的档案库列表。

``` screen
deb http://deb.debian.org/debian/ trixie main non-free-firmware contrib non-free
deb http://security.debian.org/debian-security trixie-security main non-free-firmware contrib non-free
deb http://deb.debian.org/debian/ trixie-updates main non-free-firmware contrib non-free
deb http://deb.debian.org/debian/ trixie-backports main non-free-firmware contrib non-free
```

并不需要在 \"`/etc/apt/preferences`{.literal}\" 文件中显式设置Pin-Priority值. 当新的包可用时，默认配置提供了更合理的更新 (请见 [第 2.5.3 节 "档案库层的"Release"文件"](ch02.zh-cn.html#_archive_level_release_files "2.5.3. 档案库层的“Release”文件"){.xref}).

::: itemizedlist
-   所有已安装的旧软件包都可以通过 `trixie-updates`{.literal} 档案库升级到新软件包。

-   只有从 `trixie-backports`{.literal} 档案库中手动安装的旧软件包才会通过 `trixie-backports`{.literal} 档案库升级到新软件包。
:::

当你想要从 `trixie-backports`{.literal} 档案库中手动的安装一个名叫 \"[*`package-name`{.literal}*]{.emphasis}\" 的软件及其依赖包的时候，你应该在目标档案库之前加一个 \"`-t`{.literal}\" 参数。

``` screen
$ sudo apt-get install -t trixie-backports package-name
```

::: {.warning style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[警告\]](images/warning.png)  警告
                                    不要从 [backports.debian.org](https://backports.debian.org){.ulink} 档案库安装太多软件包。它能够造成软件包依赖复杂。替代解决方案参见 [第 2.1.11 节 "怎样和不一致的要求协作"](ch02.zh-cn.html#_how_to_cope_with_conflicting_requirements "2.1.11. 怎样和不一致的要求协作"){.xref}。

### []{#_packages_from_mixed_source_of_archives}2.7.6. 不使用 [**apt-pinning**]{.strong} 的混合源档案库软件包 {#不使用-apt-pinning-的混合源档案库软件包 .title}

</div>

</div>
:::::

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    从混合源档案库中安装软件包是不被 Debian 官方发行版所支持的，除了官方支持的档案库的特殊组合以外，例如 `stable`{.literal} 的 [security updates](https://www.debian.org/security/){.ulink} 和 [stable-updates](https://wiki.debian.org/StableUpdates){.ulink}。
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

这里有一个列子，在原有只跟踪 `testing`{.literal} 的场景，操作包含在 `unstable`{.literal} 里发现的新的上游软件包版本。

::: orderedlist
1.  临时更改 \"`/etc/apt/sources.list`{.literal}\" 文件，使之指向单一的 \"`unstable`{.literal}\" 发行版路径。

2.  运行 \"`aptitude update`{.literal}\" 命令。

3.  运行 \"`aptitude install `{.literal}[*`package-name`{.literal}*]{.emphasis}\" 命令。

4.  恢复到原始 \"`/etc/apt/sources.list`{.literal}\" 文件，使之指向 `testing`{.literal} 路径。

5.  运行 \"`aptitude update`{.literal}\" 命令。
:::

使用这个手工方法，你不需要创建 \"`/etc/apt/preferences`{.literal}\" 文件，也不需要担心 [**apt-pinning**]{.strong}。但这个方法仍然是非常麻烦的。

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    当使用混合档案源的时候，因为 Debian 不会确保软件之间的兼容性，所以你必须自己去解决兼容性问题。如果软件之间存在不兼容性，系统可能会损坏。你必须能够判断这些操作所需的技术要求。使用任意混合的档案源是完全可选的操作，我并不鼓励你去使用它。
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

从不同的档案库中安装软件包的一般规则如下。

::: itemizedlist
-   非二进制软件包 (\"`Architecture: all`{.literal}\") 的安装是[**更保险的**]{.strong}。

    ::: itemizedlist
    -   文档软件包：没有特别的要求

    -   解释程序的软件包：兼容的解释器必须是可用的
    :::

-   二进制软件包 (non \"`Architecture: all`{.literal}\")通常会面临很多障碍，它的安装[**不保险的**]{.strong}。

    ::: itemizedlist
    -   库文件版本的兼容性（包括 \"`libc`{.literal}\"）

    -   与之相关的有用的程序版本的兼容性

    -   内核 [ABI](https://zh.wikipedia.org/wiki/应用程序二进制接口){.ulink} 的兼容性

    -   C++ [ABI](https://zh.wikipedia.org/wiki/应用程序二进制接口){.ulink} 的兼容性

    -   ...
    :::
:::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 为了使软件包的安装变得[**更保险**]{.strong} ，一些商业的非自由的二进制程序包可能会提供完整的静态链接库。你还是应该检查 [ABI](https://zh.wikipedia.org/wiki/应用程序二进制接口){.ulink} 的兼容性问题等等。
  ------------------------------ -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

  ------------------------------ ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 除非为了短期避免破坏软件包，从非 Debian 档案库安装二进制软件包通常是一个坏的主意。你需要查找所有存在的和你目前 Debian 系统兼容的安全技术替代方案。（参见 [第 2.1.11 节 "怎样和不一致的要求协作"](ch02.zh-cn.html#_how_to_cope_with_conflicting_requirements "2.1.11. 怎样和不一致的要求协作"){.xref}）。

### []{#_tweaking_candidate_version}2.7.7. 使用 [**apt-pinning**]{.strong} 调整获选版本 {#使用-apt-pinning-调整获选版本 .title}

</div>

</div>
:::::

::: {.warning style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------
   ![\[警告\]](images/warning.png)  警告
                                    新手用 [**apt-pinning**]{.strong} 命令肯定会造成比较大的问题。你必须避免使用这个技术，除非确实需要它。
  --------------------------------- --------------------------------------------------------------------------------------------------------
:::

没有 \"`/etc/apt/preferences`{.literal}\" 文件，APT 系统使用版本字符串来选择最新的可用版本作为 [**候选版本**]{.strong}。这是通常的状态，也是 APT 系统最推荐的使用方法。所有官方支持的档案库集合，并不要求 \"`/etc/apt/preferences`{.literal}\" 文件，因此，一些不应当被作为自动更新源的软件包，被标记为 [**NotAutomatic**]{.strong}，并被适当处理。

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                版本字符串的比较规则可以被验证，例子如下，\"`dpkg --compare-versions ver1.1 gt ver1.1~1; echo $?`{.literal}\" (参见 `dpkg`{.literal}(1))。
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------
:::

如果经常从混合源档案库中安装软件包 (参见[第 2.7.6 节 "不使用 [**apt-pinning**]{.strong} 的混合源档案库软件包"](ch02.zh-cn.html#_packages_from_mixed_source_of_archives "2.7.6. 不使用 apt-pinning 的混合源档案库软件包"){.xref}), 你可以通过创建 \"`/etc/apt/preferences`{.literal}\" 文件并且在其中写入关于调整[**候选版本**]{.strong}的软件包选取规则的合适条目 (如`apt_preferences`{.literal}(5) 中所示)来自动化这些复杂的操作。这被称为 [**apt-pinning**]{.strong}。

当使用 [**apt-pinning**]{.strong} 命令时，因为 Debian 不会确保软件之间的兼容性，所以你必须自己确认其兼容性。[**apt-pinning**]{.strong} 是完全可选的操作，我并不建议去使用它。

档案库层级的 Release 文件 (参见 [第 2.5.3 节 "档案库层的"Release"文件"](ch02.zh-cn.html#_archive_level_release_files "2.5.3. 档案库层的“Release”文件"){.xref}) 使用 `apt_preferences`{.literal}(5) 的规则.对于 [Debian 通用档案库](http://deb.debian.org/debian/dists/){.ulink} 和 [Debian 安全档案库](http://security.debian.org/debian-security/dists/){.ulink}，[**apt-pinning**]{.strong} 只在 \"suite\" 名下工作。(这点和 [Ubuntu](https://www.ubuntu.com/){.ulink} 档案库不同.)例如，你在 \"`/etc/apt/preferences`{.literal}\" 文件里面，可以使用\"`Pin: release a=unstable`{.literal}\" ，但不能使用 \"`Pin: release a=sid`{.literal}\".

当使用非 Debian 的档案库作为 [**apt-pinning**]{.strong} 的一部分时，你应该检查它们的用途和可信度。例如，Ubuntu 和 Debian 是不能混在一起的。

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 即使不创建 \"`/etc/apt/preferences`{.literal}\" 文件，在不用 [**apt-pinning**]{.strong} 命令的情况下，你也可以进行相当复杂的系统工作 (参见[第 2.6.3 节 "使用 dpkg 命令进行救援"](ch02.zh-cn.html#_rescue_with_the_dpkg_command "2.6.3. 使用 dpkg 命令进行救援"){.xref}和[第 2.7.6 节 "不使用 [**apt-pinning**]{.strong} 的混合源档案库软件包"](ch02.zh-cn.html#_packages_from_mixed_source_of_archives "2.7.6. 不使用 apt-pinning 的混合源档案库软件包"){.xref})。
  ------------------------------ --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

如下是关于 [**apt-pinning**]{.strong} 技术的简化说明。

可用的软件包源在 \"`/etc/apt/sources.list`{.literal}\" 文件里面定义，APT 系统从可用的软件包源里面选择 Pin-Priority 值最大的，作为[**升级**]{.strong} 软件包的[**候选版本**]{.strong}。如果一个软件包的 Pin-Priority 大于 1000，这个版本限制为只能 [**升级**]{.strong}，关闭了软件包降级功能（参见 [第 2.7.11 节 "紧急降级"](ch02.zh-cn.html#_emergency_downgrading "2.7.11. 紧急降级"){.xref}）。

每个软件包的 Pin-Priority 值是在 \"`/etc/apt/preferences`{.literal}\" 文件中的 \"Pin-Priority\" 条目中定义或者是使用它的默认值。

:::: table
[]{#listofnotablepinpinningtechnique}

**表 2.18. 用于 [apt-pinning]{.strong} 技术的值得注意的 Pin-Priority 值列表。**

::: table-contents
  Pin-Priority   [**apt-pinning**]{.strong} 对软件包的影响
  -------------- ----------------------------------------------------------------------------------------
  1001           安装该软件包，即使是一个降级软件包的指令
  990            用作[**目标发行版**]{.strong}档案库的默认值
  500            用作[**常规**]{.strong}档案库的默认值
  100            用于 [**NotAutomatic**]{.strong} 和 [**ButAutomaticUpgrades**]{.strong} 档案库的默认值
  100            用于[**已安装**]{.strong}软件包
  1              用于 [**NotAutomatic**]{.strong} 档案库的默认值
  -1             即使被推荐，[**也绝不安装**]{.strong}这个软件包
:::
::::

\

[**目标版本**]{.strong}档案仓库，能够由命令行选项设置，例如: \"`apt-get install -t testing some-package`{.literal}\"

[**NotAutomatic**]{.strong} 和 [**ButAutomaticUpgrades**]{.strong} 的档案是由档案库服务器上档案层级的 Release 文件来设置，(参见 [第 2.5.3 节 "档案库层的"Release"文件"](ch02.zh-cn.html#_archive_level_release_files "2.5.3. 档案库层的“Release”文件"){.xref})，同时包含\"`NotAutomatic: yes`{.literal}\" 和 \"`ButAutomaticUpgrades: yes`{.literal}\".而 [**NotAutomatic**]{.strong} 档案也是由档案库服务器上的档案层级的 Release 文件来设置，但只包含 \"`NotAutomatic: yes`{.literal}\".

来自多个档案源的[*软件包*]{.emphasis}的 [**apt-pinning 情况**]{.strong}可以通过 \"`apt-cache policy `{.literal}[*`package`{.literal}*]{.emphasis}\" 命令显示。

::: itemizedlist
-   \"`Package pin:`{.literal}\" 开头的行，列出了软件包版本的 [**pin**]{.strong} ，如果 [*package*]{.emphasis} 相关的 pin 已经定义, 例如, \"`Package pin: 0.190`{.literal}\".

-   没有 \"`Package pin:`{.literal}\" 的行存在，如果没有 [*package*]{.emphasis} 相关的定义。

-   与 [*package*]{.emphasis} 相关的 Pin-Priority 值列在所有版本字符串的右边，比如，\"`0.181 700`{.literal}\".

-   \"`0`{.literal}\" 是列在所有版本字符串的右边，如果没有 [*package*]{.emphasis} 相关的定义。例如, \"`0.181 0`{.literal}\".

-   档案库 (在 \"`/etc/apt/preferences`{.literal}\" 文件作为\"`Package: *`{.literal}\"定义) 的 Pin-Priority 值，列在所有档案库路径的左边，例如，\"`100 http://deb.debian.org/debian/ trixie-backports/main Packages`{.literal}\".
:::

### []{#_tracking_literal_testing_literal_with_some_packages_from_literal_unstable_literal}2.7.9. 使用带有 `unstable`{.literal} 软件包的 `testing`{.literal} 版本 {#使用带有-unstable-软件包的-testing-版本 .title}

</div>

</div>
:::::

::: {.warning style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------
   ![\[警告\]](images/warning.png)  警告
                                    新手用 [**apt-pinning**]{.strong} 命令肯定会造成比较大的问题。你必须避免使用这个技术，除非确实需要它。
  --------------------------------- --------------------------------------------------------------------------------------------------------
:::

如下是一个关于 [**apt-pinning**]{.strong} 技术的例子，当使用 `testing`{.literal}的时候，实现 `unstable`{.literal} 中的特定的较新的上游版本软件包的日常升级。你应该按如下所示的在 \"`/etc/apt/sources.list`{.literal}\" 文件中列出所有需要的档案库。

``` screen
deb http://deb.debian.org/debian/ testing main contrib non-free
deb http://deb.debian.org/debian/ unstable main contrib non-free
deb http://security.debian.org/debian-security testing-security main contrib
```

按如下所示的设置 \"`/etc/apt/preferences`{.literal}\" 文件。

``` screen
Package: *
Pin: release a=unstable
Pin-Priority: 100
```

当想要在此配置下从 `unstable`{.literal} 档案库中安装 \"[*`package-name`{.literal}*]{.emphasis}\" 软件及它的依赖包时，你执行带有 \"`-t`{.literal}\" 选项 (`unstable`{.literal} 的 Pin-Priority 值变为 990) 的转换目标发行版的命令。

``` screen
$ sudo apt-get install -t unstable package-name
```

在此配置下，执行 \"`apt-get update`{.literal}\" 和 \"`apt-get dist-upgrade`{.literal}\"(或者 \"`aptitude safe-upgrade`{.literal}\" 和 \"`aptitude full-upgrade`{.literal}\") 命令,会从 `testing`{.literal} 档案库升级那些从 `testing`{.literal} 档案库安装的软件包并且从 `unstable`{.literal} 档案库升级那些从 `unstable`{.literal} 档案库中安装的软件包。

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    小心不要去移除 \"`/etc/apt/sources.list`{.literal}\" 文件中的 \"`testing`{.literal}\" 档案库。如果文件中没有 \"`testing`{.literal}\" ，APT 系统会使用更加新的 `unstable`{.literal} 档案库升级软件包。
  --------------------------------- -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                我通常会在上述操作后，马上注释掉 \"`/etc/apt/sources.list`{.literal}\" 文件中的 \"`unstable`{.literal}\" 档案库记录。这避免了因为处理 \"`/etc/apt/sources.list`{.literal}\" 文件中的众多记录而造成的升级缓慢虽然同时也阻止了那些从 `unstable`{.literal} 档案库中安装的软件包通过 `unstable`{.literal} 升级。
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}
  ----------------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                如果 \"`/etc/apt/preferences`{.literal}\" 文件中 \"`Pin-Priority: 1`{.literal}\" 替代了 \"`Pin-Priority:100`{.literal}\"，即使 \"`/etc/apt/sources.list`{.literal}\" 文件中的 \"`testing`{.literal}\" 记录被删除了，Pin-Priority 值为 100 的已安装软件包也不会通过 `unstable`{.literal} 档案库升级。
  ----------------------------- ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

如果你希望自动跟踪 `unstable`{.literal} 里某些特殊的软件包，而在安装时不再使用初始化选项 \"`-t unstable`{.literal}\" ， 你必须创建 \"`/etc/apt/preferences`{.literal}\" 文件，并在该文件顶部按下面的方式清晰的列出所有那些软件包。

``` screen
Package: package-1
Pin: release a=unstable
Pin-Priority: 700

Package: package-2
Pin: release a=unstable
Pin-Priority: 700
```

如下是为每个特定的软件包设置 Pin-Priority 值。例如，为了使用最新的 `unstable`{.literal} 的英文版 \"Debian Reference\"，你应该在 \"`/etc/apt/preferences`{.literal}\" 文件中写入以下条目。

``` screen
Package: debian-reference-en
Pin: release a=unstable
Pin-Priority: 700

Package: debian-reference-common
Pin: release a=unstable
Pin-Priority: 700
```

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}
  ----------------------------- -------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                即使你使用的是 `stable`{.literal} 档案库，[**apt-pinning**]{.strong} 技术仍然是有效的。根据我以前的经验，从 `unstable`{.literal} 档案库安装的文档包一直是安全的。
  ----------------------------- -------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

### []{#_tracking_literal_unstable_literal_with_some_packages_from_literal_experimental_literal}2.7.10. 使用带有 `experimental`{.literal} 软件包的 `unstable`{.literal} 版本 {#使用带有-experimental-软件包的-unstable-版本 .title}

</div>

</div>
:::::

::: {.warning style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------
   ![\[警告\]](images/warning.png)  警告
                                    新手用 [**apt-pinning**]{.strong} 命令肯定会造成比较大的问题。你必须避免使用这个技术，除非确实需要它。
  --------------------------------- --------------------------------------------------------------------------------------------------------
:::

这是使用[**apt-pinning**]{.strong} 的另一个示例，该示例主要使用`unstable`{.literal}源，但包含了`experimental`{.literal}源, 该源可用于安装上游更新的软件包 . 需要包含在\"`/etc/apt/sources.list`{.literal}\" 文件中的列表如下。

``` screen
deb http://deb.debian.org/debian/ unstable main contrib non-free
deb http://deb.debian.org/debian/ experimental main contrib non-free
deb http://security.debian.org/ testing-security main contrib
```

由于`experimental`{.literal}源是 [**非自动（NotAutomatic）**]{.strong}的源 (参见 [第 2.5.3 节 "档案库层的"Release"文件"](ch02.zh-cn.html#_archive_level_release_files "2.5.3. 档案库层的“Release”文件"){.xref})，其默认的Pin-Priority值 被设置为1 (\<\<100) . 并不需要在\"`/etc/apt/preferences`{.literal}\"文件中设置Pin-Priority值，只需要指定 `experimental`{.literal} 源，除非你需要在下次更新时自动升级时更新特定软件包.

### []{#_emergency_downgrading}2.7.11. 紧急降级 {#紧急降级 .title}

</div>

</div>
:::::

::: {.warning style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------
   ![\[警告\]](images/warning.png)  警告
                                    新手用 [**apt-pinning**]{.strong} 命令肯定会造成比较大的问题。你必须避免使用这个技术，除非确实需要它。
  --------------------------------- --------------------------------------------------------------------------------------------------------
:::

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    降级在 Debian 设计上就不被官方支持。仅仅是在紧急恢复过程中需要做的一部分工作。尽管憎恨这种情形，但降级在很多场景下工作得也不错。对于重要系统，你应当在恢复操作后备份所有重要数据，并从零开始重新安装一个新的系统。
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

你可以通过控制[**候选版本**]{.strong}从新的档案库降级到旧的档案库（参见 [第 2.7.7 节 "使用 [**apt-pinning**]{.strong} 调整获选版本"](ch02.zh-cn.html#_tweaking_candidate_version "2.7.7. 使用 apt-pinning 调整获选版本"){.xref}），从而使损坏的系统恢复。下面是一种懒惰的方法，可以避免许多冗长的 "`dpkg -i `{.literal}[*`broken-package`{.literal}*]{.emphasis}`_`{.literal}[*`old-version`{.literal}*]{.emphasis}`.deb`{.literal}" 命令（参见 [第 2.6.3 节 "使用 dpkg 命令进行救援"](ch02.zh-cn.html#_rescue_with_the_dpkg_command "2.6.3. 使用 dpkg 命令进行救援"){.xref}）。

搜索 "`/etc/apt/sources.list`{.literal}" 文件中像下面那样使用 `unstable`{.literal} 的行。

``` screen
deb http://deb.debian.org/debian/ sid main contrib non-free
```

使用下面的行替换它，从而改为使用 `testing`{.literal}。

``` screen
deb http://deb.debian.org/debian/ forky main contrib non-free
```

按如下所示的设置 \"`/etc/apt/preferences`{.literal}\" 文件。

``` screen
Package: *
Pin: release a=testing
Pin-Priority: 1010
```

运行 "`apt-get update; apt-get dist-upgrade`{.literal}" 使整个系统的软件包强制降级。

在紧急降级后，移除 "`/etc/apt/preferences`{.literal}" 这个特殊的文件。

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[提示\]](images/tip.png)  提示
                                这是一个好方法，移除（不是清除！）尽可能多地软件包，来减少依赖问题。你可能需要手动移除和安装一些软件包来使系统降级。需要特别注意 Linux 内核、引导程序、udev、PAM、APT 和网络相关的软件包以及它们的配置文件。
  ----------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

### []{#_the_equivs_package}2.7.12. equivs 软件包 {#equivs-软件包 .title}

</div>

</div>
:::::

如果你从源代码编译了一个程序来代替 Debian 软件包，最好将它做成一个真正的本地 Debian 软件包（`*.deb`{.literal}）并使用私人档案库。

如果你选择从源代码编译一个程序并将它安装到 "`/usr/local`{.literal}"，你可能需要使用 `equivs`{.literal} 作为最后步骤来满足缺失的软件包依赖。

``` screen
Package: equivs
Priority: optional
Section: admin
Description: Circumventing Debian package dependencies
 This package provides a tool to create trivial Debian packages.
 Typically these packages contain only dependency information, but they
 can also include normal installed files like other packages do.
 .
 One use for this is to create a metapackage: a package whose sole
 purpose is to declare dependencies and conflicts on other packages so
 that these will be automatically installed, upgraded, or removed.
 .
 Another use is to circumvent dependency checking: by letting dpkg
 think a particular package name and version is installed when it
 isn't, you can work around bugs in other packages' dependencies.
 (Please do still file such bugs, though.)
```

### []{#_porting_a_package_to_the_stable_system}2.7.13. 移植一个软件包到 stable 系统 {#移植一个软件包到-stable-系统 .title}

</div>

</div>
:::::

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------
   ![\[小心\]](images/caution.png)  小心
                                    由于系统差异，不能够保证这里描述的过程能够工作，而不需要额外的手工处理。
  --------------------------------- --------------------------------------------------------------------------
:::

对于部分升级的 `stable`{.literal} 系统，使用源软件包在运行环境中重新构建一个软件包是不错的选择。这可以避免因为依赖关系导致大量软件包升级。

在 `stable`{.literal} 系统的 "`/etc/apt/sources.list`{.literal}" 文件中添加下列条目。

``` screen
deb-src http://deb.debian.org/debian unstable  main contrib non-free
```

如下安装编译所需的软件包并下载源软件包。

``` screen
# apt-get update
# apt-get dist-upgrade
# apt-get install fakeroot devscripts build-essential
# apt-get build-dep foo
$ apt-get source foo
$ cd foo*
```

如果需要向后移植，可以从 backport 的软件包中更新一些工具链软件包，例如 `dpkg`{.literal} 和 `debhelper`{.literal}。

执行下列命令。

``` screen
$ dch -i
```

更新软件包版本，例如在 "`debian/changelog`{.literal}" 中附加一个 "`+bp1`{.literal}"

像下面那样构建软件包并将它们安装到系统中。

``` screen
$ debuild
$ cd ..
# debi foo*.changes
```
