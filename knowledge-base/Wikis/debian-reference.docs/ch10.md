### []{#_idioms_for_the_archive}10.1.3. 归档语法 {#归档语法 .title}

</div>

</div>
:::::

以下是用不同的工具压缩和解压缩整个 \"`./source`{.literal}\" 目录中的内容。

GNU `tar`{.literal}(1):

``` screen
$ tar -cvJf archive.tar.xz ./source
$ tar -xvJf archive.tar.xz
```

或者，如下所示。

``` screen
$ find ./source -xdev -print0 | tar -cvJf archive.tar.xz --null -T -
```

`cpio`{.literal}(1):

``` screen
$ find ./source -xdev -print0 | cpio -ov --null > archive.cpio; xz archive.cpio
$ zcat archive.cpio.xz | cpio -i
```

### []{#_idioms_for_the_copy}10.1.4. 复制语法 {#复制语法 .title}

</div>

</div>
:::::

如下是用不同的工具复制整个 \"`./source`{.literal}\" 目录中的内容。

::: itemizedlist
-   本地复制: \"`./source`{.literal}\" 目录 → \"`/dest`{.literal}\" 目录

-   远程复制：本地主机上的 \"`./source" 目录`{.literal} → \"`user@host.dom`{.literal}\" 主机上的 \"`/dest`{.literal}\" 目录
:::

`rsync`{.literal}(8):

``` screen
# cd ./source; rsync -aHAXSv . /dest
# cd ./source; rsync -aHAXSv . user@host.dom:/dest
```

你能够选择使用"源目录上的反斜杠"语法。

``` screen
# rsync -aHAXSv ./source/ /dest
# rsync -aHAXSv ./source/ user@host.dom:/dest
```

或者，如下所示。

``` screen
# cd ./source; find . -print0 | rsync -aHAXSv0 --files-from=- . /dest
# cd ./source; find . -print0 | rsync -aHAXSv0 --files-from=- . user@host.dom:/dest
```

GNU `cp`{.literal}(1) 和 openSSH `scp`{.literal}(1):

``` screen
# cd ./source; cp -a . /dest
# cd ./source; scp -pr . user@host.dom:/dest
```

GNU `tar`{.literal}(1):

``` screen
# (cd ./source && tar cf - . ) | (cd /dest && tar xvfp - )
# (cd ./source && tar cf - . ) | ssh user@host.dom '(cd /dest && tar xvfp - )'
```

`cpio`{.literal}(1):

``` screen
# cd ./source; find . -print0 | cpio -pvdm --null --sparse /dest
```

你能够在所有包含 \"`.`{.literal}\" 的例子里用 \"`foo`{.literal}\" 替代 \"`.`{.literal}\"，这样就可以从 \"`./source/foo`{.literal}\" 目录复制文件到 \"`/dest/foo`{.literal}\" 目录。

在所有包含 \"`.`{.literal}\" 的列子里，你能够使用绝对路径 \"`/path/to/source/foo`{.literal}\" 来代替 \"`.`{.literal}\"，这样可以去掉 \"`cd ./source;`{.literal}\". 如下所示，这些文件会根据工具的不同，拷贝到不同的位置。

::: itemizedlist
-   \"`/dest/foo`{.literal}\": `rsync`{.literal}(8), GNU `cp`{.literal}(1), 和 `scp`{.literal}(1)

-   \"`/dest/path/to/source/foo`{.literal}\": GNU `tar`{.literal}(1), 和 `cpio`{.literal}(1)
:::

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}

### []{#_idioms_for_the_selection_of_files}10.1.5. 查找文件的语法 {#查找文件的语法 .title}

</div>

</div>
:::::

`find`{.literal}(1) 被用作从归档中筛选文件也被用作拷贝命令 (参见[第 10.1.3 节 "归档语法"](ch10.zh-cn.html#_idioms_for_the_archive "10.1.3. 归档语法"){.xref}和[第 10.1.4 节 "复制语法"](ch10.zh-cn.html#_idioms_for_the_copy "10.1.4. 复制语法"){.xref}) 或者用于 `xargs`{.literal}(1) (参见[第 9.4.9 节 "使用文件循环来重复一个命令"](ch09.zh-cn.html#_repeating_a_command_looping_over_files "9.4.9. 使用文件循环来重复一个命令"){.xref})。通过 find 的命令行参数能够使其功能得到加强。

以下是 `find`{.literal}(1)基本语法的总结。

::: itemizedlist
-   find 条件参数的运算规则是从左到右。

-   一旦输出是确定的，那么运算就会停止。

-   "逻辑 [**OR**]{.strong}\" （由条件之间的 \"`-o`{.literal}\" 参数指定的）优先级低于 \"逻辑 [**AND**]{.strong}\" （由 \"`-a`{.literal}\" 参数指定或者条件之间没有任何参数）。

-   "逻辑 [**NOT**]{.strong}\" (由条件前面的 \"`!`{.literal}\" 指定) 优先级高于 "逻辑 [**AND**]{.strong}"。

-   \"`-prune`{.literal}\" 总是返回逻辑 [**TRUE**]{.strong} 并且如果这个目录是存在的，将会搜索除这个目录以外的文件。

-   \"`-name`{.literal}\" 选项匹配带有 shell 通配符 (参见[第 1.5.6 节 "Shell 通配符"](ch01.zh-cn.html#_shell_glob "1.5.6. Shell 通配符"){.xref}) 的文件名但也匹配带有类似 \"`*`{.literal}\" 和 \"`?`{.literal}\" 元字符的 `.`{.literal}\"。(新的 [POSIX](https://zh.wikipedia.org/wiki/POSIX){.ulink} 特性)

-   \"`-regex`{.literal}\" 匹配整个文件路径，默认采用 emacs 风格的 [**BRE**]{.strong} (参见[第 1.6.2 节 "正则表达式"](ch01.zh-cn.html#_regular_expressions "1.6.2. 正则表达式"){.xref})。

-   \"`-size`{.literal}\" 根据文件大小来匹配 (值前面带有 \"`+`{.literal}\" 号匹配更大的文件，值前面带有 \"`-`{.literal}\" 号匹配更小的文件)

-   \"`-newer`{.literal}\" 参数匹配比参数名中指定的文件还要新的文件。

-   \"`-print0`{.literal}\" 参数总是返回逻辑 [**TRUE**]{.strong} 并将完整文件名 ([null terminated](https://zh.wikipedia.org/wiki/Null_character){.ulink}) 打印到标准输出设备上。
:::

如下是 `find`{.literal}(1) 语法格式。

``` screen
# find /path/to \
    -xdev -regextype posix-extended \
    -type f -regex ".*\.cpio|.*~" -prune -o \
    -type d -regex ".*/\.git" -prune -o \
    -type f -size +99M -prune -o \
    -type f -newer /path/to/timestamp -print0
```

这些命令会执行如下动作。

::: orderedlist
1.  查找 \"`/path/to`{.literal}\" 下的所有文件

2.  限定全局查找的文件系统并且使用的是 [**ERE**]{.strong} (参见[第 1.6.2 节 "正则表达式"](ch01.zh-cn.html#_regular_expressions "1.6.2. 正则表达式"){.xref})

3.  通过停止处理的方式来排除匹配 \"`.*\.cpio`{.literal}\" 或 \"`.*~`{.literal}\" 正则表达式的文件

4.  通过停止处理的方式来排除匹配 \"`.*/\.git`{.literal}\" 正则表达式的目录

5.  通过停止处理的方式来排除比 99MB (1048576 字节单元) 更大的文件

6.  显示文件名，满足以上搜索条件并且比 \"`/path/to/timestamp`{.literal}\" 新的文件
:::

请留心以上例子中的 \"`-prune -o`{.literal}\" 排除文件的习惯用法。

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ -----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 对于非 Debian 系的 [Unix-like](https://zh.wikipedia.org/wiki/Unix-like){.ulink} 系统，有些参数可能不被 `find`{.literal}(1) 命令所支持。在这种情况下，应该考虑调整匹配方法并用 \"`-print`{.literal}\" 替代 \"`-print0`{.literal}\"。你可能同样需要更改其他相关的命令。

### []{#_backup_and_recovery_policy}10.2.1. 备份和恢复策略 {#备份和恢复策略 .title}

</div>

</div>
:::::

有 3 个关键的因素决定实际的备份和恢复策略。

::: orderedlist
1.  知道要备份和恢复什么。

    ::: itemizedlist
    -   你自己创建的数据文件：在 \"`~/`{.literal}\" 下的数据

    -   你使用的应用程序创建的数据文件：在 \"`/var/`{.literal}\" 下的数据（除了 \"`/var/cache/`{.literal}\"，\"`/var/run/`{.literal}\" 和 \"`/var/tmp/`{.literal}\")

    -   系统配置文件：在 \"`/etc/`{.literal}" 下的数据

    -   本地程序：在 \"`/usr/local/`{.literal}\" 或 \"`/opt/`{.literal}\" 下的数据

    -   系统安装信息：关键步骤 (分区,\...) 的纯文本备忘录

    -   验证数据结果：通过实验性的恢复操作来预先验证

        ::: itemizedlist
        -   用户进程的 Cron 工作，文件在 \"`/var/spool/cron/crontabs`{.literal}\" 目录，并且重启 `cron`{.literal}(8)。参见[第 9.4.14 节 "定时任务安排"](ch09.zh-cn.html#_scheduling_tasks_regularly "9.4.14. 定时任务安排"){.xref}来获得关于 `cron`{.literal}(8) 和 `crontab`{.literal}(1) 的信息。

        -   用户进程的 Systemd 计时器工作：文件在 \"`~/.config/systemd/user`{.literal}\" 目录。参见 `systemd.timer`{.literal}(5) 和 `systemd.service`{.literal}(5)。

        -   用户进程的自动启动工作：文件在 \"`~/.config/autostart`{.literal}\" 目录。参见 [Desktop Application Autostart Specification](https://specifications.freedesktop.org/autostart-spec/autostart-spec-latest.html){.ulink}。
        :::
    :::

2.  知道怎样去备份和恢复。

    ::: itemizedlist
    -   安全的数据存储：保护其免于覆盖和系统故障

    -   经常备份：有计划的备份

    -   冗余备份：数据镜像

    -   傻瓜式操作：单个简单命令备份
    :::

3.  评估涉及的风险和成本。

    ::: itemizedlist
    -   数据丢失的风险

        ::: itemizedlist
        -   数据至少是应该在不同的磁盘分区上，最好是在不同的磁盘和机器上，来承受文件系统发生的损坏。重要数据最好存储在一个只读文件系统上。[^\[4\]^](#ftn.id20250){#id20250 .footnote}
        :::

    -   数据非法访问的风险

        ::: itemizedlist
        -   敏感的身份数据，比如 \"`/etc/ssh/ssh_host_*_key`{.literal}\", \"`~/.gnupg/*`{.literal}\", \"`~/.ssh/*`{.literal}\", \"`~/.local/share/keyrings/*`{.literal}\", \"`/etc/passwd`{.literal}\", \"`/etc/shadow`{.literal}\", \"`popularity-contest.conf`{.literal}\", \"`/etc/ppp/pap-secrets`{.literal}\", and \"`/etc/exim4/passwd.client`{.literal}\" 应当使用加密备份。[^\[5\]^](#ftn.id20267){#id20267 .footnote} (参见 [第 9.9 节 "数据加密提示"](ch09.zh-cn.html#_data_encryption_tips "9.9. 数据加密提示"){.xref}。)

        -   即使在信任的系统上，也不能够硬编码系统登录密码或者加密密码到任何脚本里面。(参见 [第 10.3.6 节 "密码密钥环"](ch10.zh-cn.html#_password_keyring "10.3.6. 密码密钥环"){.xref}。)
        :::

    -   数据丢失的方式及其可能性

        ::: itemizedlist
        -   硬件（特别是硬盘）将会损坏

        -   文件系统可能会损坏，里面的数据可能被丢失

        -   对违规安全访问而言，远程存储系统不能够被信任

        -   弱的密码保护能够被轻松的破解

        -   文件权限系统可以被破解
        :::

    -   备份所需的资源：人力，硬件，软件，...

        ::: itemizedlist
        -   使用 cron 任务或者 systemd 计时器任务来自动化调度备份工作
        :::
    :::
:::

### []{#_backup_tips}10.2.3. 备份技巧 {#备份技巧 .title}

</div>

</div>
:::::

对于一个个人工作站，为服务器环境设计的全功能备份套件工具也行不是最合适的。与此同时，已有的用于工作站的备份工具有一些不足。

这里有一些技巧让备份更加容易，只需用户做最小的工作。这些技巧可以同任意备份工具一起使用。

出于演示的目的，让我们假设基本用户和组名为 `penguin`{.literal}，创建一个备份和快照脚本例子\"`/usr/local/bin/bkss.sh`{.literal}\"：

``` screen
#!/bin/sh -e
SRC="$1" # source data path
DSTFS="$2" # backup destination filesystem path
DSTSV="$3" # backup destination subvolume name
DSTSS="${DSTFS}/${DSTSV}-snapshot" # snapshot destination path
if [ "$(stat -f -c %T "$DSTFS")" != "btrfs" ]; then
  echo "E: $DESTFS needs to be formatted to btrfs" >&2 ; exit 1
fi
MSGID=$(notify-send -p "bkup.sh $DSTSV" "in progress ...")
if [ ! -d "$DSTFS/$DSTSV" ]; then
  btrfs subvolume create "$DSTFS/$DSTSV"
  mkdir -p "$DSTSS"
fi
rsync -aHxS --delete --mkpath "${SRC}/" "${DSTFS}/${DSTSV}"
btrfs subvolume snapshot -r "${DSTFS}/${DSTSV}" ${DSTSS}/$(date -u --iso=min)
notify-send -r "$MSGID" "bkup.sh $DSTSV" "finished!"
```

这里，只使用基本工具 `rsync`{.literal}(1)来帮助备份，存储空间使用 [Btrfs](https://zh.wikipedia.org/wiki/Btrfs){.ulink} 来高效利用。

#### []{#_mount_event_triggered_backup}10.2.3.2. 挂载事件触发的备份 {#挂载事件触发的备份 .title}

     $ systemctl --user enable bkup-BKUP.service

#### []{#_timer_event_triggered_backup}10.2.3.3. 时间事件触发的备份 {#时间事件触发的备份 .title}

这里是一个由时间事件触发的自动备份例子。

::: itemizedlist
-   按 [第 10.2.3.1 节 "GUI（图形用户界面）备份"](ch10.zh-cn.html#_gui_backup "10.2.3.1. GUI（图形用户界面）备份"){.xref} 的方式准备一个 USB 存储设备用于备份。

-   创建一个如下的 systemd 时间单元文件\"`~/.config/systemd/user/snap-Documents.timer`{.literal}\"：

    [Unit]
    Description=Run btrfs subvolume snapshot on timer

    OnStartupSec=30
    OnUnitInactiveSec=900

    [Install]
    WantedBy=timers.target

-   创建一个如下的 systemd 服务单元文件\"`~/.config/systemd/user/snap-Documents.service`{.literal}\"：

    [Unit]
    Description=Run btrfs subvolume snapshot
    Documentation=man:btrfs(1)

    [Service]

    Nice=15

    IOSchedulingClass=idle
    CPUSchedulingPolicy=idle

    StandardError=append:%h/.cache/systemd-snap.log

-   用下面的方式启用这个 systemd 单元配置：

    ``` screen
     $ systemctl --user enable snap-Documents.timer
    ```
:::

对于每一次时间事件，你的数据从\"`~/Documents`{.literal}\"备份到 USB 存储设备，并创建了一个只读快照。

这里，当前 systemd 在内存中的 systemd 用户时间单元的名字，使用 \"`systemctl --user list-units --type=timer`{.literal}\" 来调用服务管理器来查询。

对于现在的桌面系统，这个 systemd 方案，比起传统的 Unix `at`{.literal}(1)、`cron`{.literal}(8) 或 `anacron`{.literal} 方式，能够提供更精致的细粒度控制。

### []{#_password_keyring}10.3.6. 密码密钥环 {#密码密钥环 .title}

</div>

</div>
:::::

在 GNOME 系统，GUI（图形用户界面）工具 `seahorse`{.literal}(1) 管理密码，安全的在密钥环 `~/.local/share/keyrings/*`{.literal} 里面保存它们。

`secret-tool`{.literal}(1) 能够从命令行存储密码到钥匙环。

让我们存储 LUKS/dm-crypt 加密磁盘镜像用到的密码

``` screen
$ secret-tool store --label='LUKS passphrase for disk.img' LUKS my_disk.img
Password: ********
```

这个存储的密码能够被获取并给到其它程序，比如 `cryptsetup`{.literal}(8)。

``` screen
$ secret-tool lookup LUKS my_disk.img | \
  cryptsetup open disk.img disk_img --type luks --keyring -
$ sudo mount /dev/mapper/disk_img /mnt
```

### []{#_configuration_of_git_client}10.5.1. 配置 Git 客户端 {#配置-git-客户端 .title}

</div>

</div>
:::::

你可以在 \"`~/.gitconfig`{.literal}\" 里面设置几个 Git 接下来需要使用的全局配置，比如说你的名字和电子邮件地址。

``` screen
$ git config --global user.name "Name Surname"
$ git config --global user.email yourname@example.com
```

你也可以按如下所示定制 Git 的默认行为。

``` screen
$ git config --global init.defaultBranch main
$ git config --global pull.rebase true
$ git config --global push.default current
```

如果你习惯使用 CVS 或 Subversion 命令，你也许希望设置如下几个命令别名。

``` screen
$ git config --global alias.ci "commit -a"
$ git config --global alias.co checkout
```

你能够通过如下方式检查你的全局配置：

``` screen
$ git config --global --list
```

### []{#_basic_git_commands}10.5.2. 基本的 Git 命令 {#基本的-git-命令 .title}

</div>

</div>
:::::

Git 操作涉及几个数据。

::: itemizedlist
-   工作树目录保持面向用户的文件，你可以对这些文件做修改。

    ::: itemizedlist
    -   需要被记录的改变，必须明确的被选择并暂存到索引。这是 `git add`{.literal} 和 `git rm`{.literal} 命令。
    :::

-   索引保持暂存文件。

    ::: itemizedlist
    -   在接下来的请求之前，暂存文件将被提交到本地仓库。这个是 `git commit`{.literal} 命令。
    :::

-   本地仓库保持已经提交的的文件。

    ::: itemizedlist
    -   Git 记录提交数据的链接历史并在仓库里面将它们作为分支组织。

    -   本地仓库通过 `git push`{.literal} 命令发送数据到远程仓库。

    -   本地仓库能够通过 `git fetch`{.literal} 和 `git pull`{.literal} 命令从远程仓库接收数据。

        ::: itemizedlist
        -   `git pull`{.literal} 命令在 `git fetch`{.literal} 后执行 `git merge`{.literal} 或 `git rebase`{.literal} 命令。

        -   这里，`git merge`{.literal} 联合两个独立分支的历史结尾到一个点。（在没有定制的 `git pull`{.literal} ，这个是默认的，同时对上游作者发布分支到许多人时，也是好的 ）

        -   这里，`git rebase`{.literal} 创建一个远程分支的序列历史的单个分支，跟着本地分支。（这是定制 `pull.rebase true`{.literal} 的情况，对我们其余的用途有用。）
        :::
    :::

-   远程仓库保持已经提交的文件。

    ::: itemizedlist
    -   到远程仓库的通信，使用安全的通信协议，比如 SSH 或 HTTPS。
    :::
:::

工作树是在 `.git/`{.literal} 目录之外的文件。在 `.git/`{.literal} 目录里面的文件，包括索引、本地仓库数据和一些 git 配置的文本文件。
