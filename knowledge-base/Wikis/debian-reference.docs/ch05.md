### []{#_the_hostname_resolution}5.1.1. 主机名解析 {#主机名解析 .title}

</div>

</div>
:::::

主机名解析，目前也是由 [NSS (名字服务转换 Name Service Switch)](https://zh.wikipedia.org/wiki/Name_Service_Switch){.ulink} 机制来支持。这个解析的流程如下。

::: orderedlist
1.  \"`/etc/nsswitch.conf`{.literal}\" 文件里的 \"`hosts: files dns`{.literal}\" 这段规定主机名解析顺序。 (代替 \"`/etc/host.conf`{.literal}\" 文件里的\"`order`{.literal}\" 这段原有的功能。)

2.  `files`{.literal} 方式首先被调用。如果主机名在 \"`/etc/hosts`{.literal}\" 文件里面发现，则返回所有有效地址并退出。 ( \"`/etc/host.conf`{.literal}\" 文件包含 \"`multi on`{.literal}\".)

3.  `dns`{.literal} 方式被调用。如果主机名通过查询 \"`/etc/resolv.conf`{.literal}\" 文件里面写的 [互联网域名系统 Domain Name System (DNS)](https://zh.wikipedia.org/wiki/Domain_Name_System){.ulink} 来找到，则返回所有有效地址并退出。
:::

一个典型的工作站在安装时就会设置主机名，例如\"[*host_name*]{.emphasis}\"和设置为空字符串的可选域名。这样，\"`/etc/hosts`{.literal}\"看起来像下文所示的这样。

``` screen
127.0.0.1 localhost
127.0.1.1 host_name

# The following lines are desirable for IPv6 capable hosts
::1     localhost ip6-localhost ip6-loopback
ff02::1 ip6-allnodes
ff02::2 ip6-allrouters
```

每一行由 [IP 地址](https://zh.wikipedia.org/wiki/IP_address){.ulink} 开始，接下来是相关联的[主机名](https://zh.wikipedia.org/wiki/Hostname){.ulink}.

在这个例子的第二行 `127.0.1.1`{.literal} IP 地址也许不会在其它类 Unix 系统发现。[Debian Installer](https://www.debian.org/devel/debian-installer/){.ulink} 为没有永久 IP 地址的系统创建这个条目，作为某些软件（如 GNOME）的一个变通方法，见文档 [bug #719621](https://bugs.debian.org/719621){.ulink}.

[*host_name*]{.emphasis} 匹配在\"`/etc/hostname`{.literal}\"里定义的主机名。(参见 [第 3.8.1 节 "主机名"](ch03.zh-cn.html#_the_hostname "3.8.1. 主机名"){.xref})。

对于有永久 IP 地址的系统，这个永久 IP 地址应当代替这里的 `127.0.1.1`{.literal}。

对于有永久 IP 地址和有 [域名系统 Domain Name System (DNS)](https://zh.wikipedia.org/wiki/Domain_Name_System){.ulink}提供[完全资格域名 fully qualified domain name (FQDN)](https://zh.wikipedia.org/wiki/FQDN){.ulink} 的系统，规范名 [*host_name*]{.emphasis}.[*domain_name*]{.emphasis} 应当被用来代替 [*host_name*]{.emphasis}.

如果 `resolvconf`{.literal} 软件包没有安装，\"`/etc/resolv.conf`{.literal}\" 是一个静态文件。如果安装了，它是一个符号链接。此外，它包含有解析策略的初始化信息。如 DNS 是 IP=\"`192.168.11.1`{.literal}\",则包含如下。

``` screen
nameserver 192.168.11.1
```

`resolvconf`{.literal} 软件包使这个 \"`/etc/resolv.conf`{.literal}\" 文件成为一个符号链接，并通过钩子脚本自动管理其内容。

对于典型 adhoc 局域网环境下的 PC 工作站，除了基本的 `files`{.literal} 和 `dns`{.literal} 方式之外，主机名还能够通过 [Multicast DNS](https://zh.wikipedia.org/wiki/Multicast_DNS){.ulink} (mDNS)进行解析。

::: itemizedlist
-   [Avahi](https://zh.wikipedia.org/wiki/Avahi_(software)){.ulink} 提供 Debian 下的组播 DNS 发现框架。

-   它和 [Apple Bonjour / Apple Rendezvous](https://zh.wikipedia.org/wiki/Bonjour){.ulink} 相当.

-   `libnss-mdns`{.literal} 插件包提供 mDNS 的主机名解析，GNU C 库 (glibc)的 GNU 名字服务转换 Name Service Switch (NSS) 功能支持 mDNS。

-   \"`/etc/nsswitch.conf`{.literal}\" 文件应当有像 \"`hosts: files mdns4_minimal [NOTFOUND=return] dns`{.literal}\" 这样的一段(其它配置参见 `/usr/share/doc/libnss-mdns/README.Debian`{.literal})。

-   一个使用 [\".local\"](https://zh.wikipedia.org/wiki/.local){.ulink} [pseudo-top-level domain](https://zh.wikipedia.org/wiki/Pseudo-top-level_domain){.ulink} 后缀的主机名解析，是用 IPv4 地址 \"`224.0.0.251`{.literal}\" 或 IPv6 地址 \"`FF02::FB`{.literal}\"发送一个组播 UDP 包的 mDNS 查询信息。
:::

  ------------------------------ --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 [域名系统 Domain Name System](https://zh.wikipedia.org/wiki/Domain_Name_System){.ulink} 中的[扩展通用顶级域名 expansion of generic Top-Level Domains (gTLD)](https://newgtlds.icann.org/en/program-status/delegated-strings){.ulink} 还在进行中。在局域网内，选择一个域名时，请提防[名字冲突 name collision](https://icannwiki.com/Name_Collision){.ulink}。
  ------------------------------ --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------

### []{#_finding_optimal_mtu}5.6.1. 找出最佳 MTU {#找出最佳-mtu .title}

</div>

</div>
:::::

网络管理器通常会自动设置最佳 [最大传输单元 (MTU)](https://zh.wikipedia.org/wiki/Maximum_transmission_unit){.ulink} 。

在一些场景中，在用 `ping`{.literal}(8) 加上\"`-M do`{.literal}\"选项发送各种大小的 ICMP 报文数据包进行实验后，你希望可以手动设置 MTU。MTU 是最大可完成没有 IP 分片的数据包大小加上 28 字节（IPv4）或 48 字节（IPv6）。下面的列子，发现 IPv4 连接的 MTU 是 1460，IPv6 连接的 MTU 是 1500。

``` screen
$ ping -4 -c 1 -s $((1500-28)) -M do www.debian.org
PING  (149.20.4.15) 1472(1500) bytes of data.
ping: local error: message too long, mtu=1460

---  ping statistics ---
1 packets transmitted, 0 received, +1 errors, 100% packet loss, time 0ms

$ ping -4 -c 1 -s $((1460-28)) -M do www.debian.org
PING  (130.89.148.77) 1432(1460) bytes of data.
1440 bytes from klecker-misc.debian.org (130.89.148.77): icmp_seq=1 ttl=50 time=325 ms

---  ping statistics ---
1 packets transmitted, 1 received, 0% packet loss, time 0ms
rtt min/avg/max/mdev = 325.318/325.318/325.318/0.000 ms
$ ping -6 -c 1 -s $((1500-48)) -M do www.debian.org
PING www.debian.org(mirror-csail.debian.org (2603:400a:ffff:bb8::801f:3e)) 1452 data bytes
1460 bytes from mirror-csail.debian.org (2603:400a:ffff:bb8::801f:3e): icmp_seq=1 ttl=47 time=191 ms

--- www.debian.org ping statistics ---
1 packets transmitted, 1 received, 0% packet loss, time 0ms
rtt min/avg/max/mdev = 191.332/191.332/191.332/0.000 ms
```

这个过程是 [路径 MTU (PMTU) 发现](https://zh.wikipedia.org/wiki/Path_MTU_discovery){.ulink} ([RFC1191](https://datatracker.ietf.org/doc/rfc1191/){.ulink}) ， `tracepath`{.literal}(8) 命令能够自动完成这个。

:::: table
[]{#basicguidelinesoeoptimalmtuvalue}

**表 5.6. 最佳 MTU 值的基本指引方法**

::: table-contents
  网络环境                       MTU    基本原理
  ------------------------------ ------ ------------
  拨号连接(IP: PPP)              576    标准的
  以太网连接 (IP: DHCP 或固定)   1500   默认标准值
:::
::::

\

除了这些基本的指引方法外，你还应当知道下面的信息。

::: itemizedlist
-   使用任何隧道方式([VPN](https://zh.wikipedia.org/wiki/Virtual_private_network){.ulink} 等.)的最佳 MTU 需要进一步减去它们上面的头部。

-   MTU 值不应当超过通过实验验证的 PMTU 值。

-   当遇到其它限制的时候，较大的 MTU 值通常比较好。
:::

[最大分片大小](https://zh.wikipedia.org/wiki/Maximum_segment_size){.ulink} (MSS) 是另外一种衡量包大小的方法。MSS 和 MTU 的关系如下.

::: itemizedlist
-   对于 IPv4， MSS = MTU - 40

-   对于 IPv6，MSS = MTU - 60
:::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 基于 `iptables`{.literal}(8) (参见 [第 5.7 节 "Netfilter 网络过滤框架"](ch05.zh-cn.html#_netfilter_infrastructure "5.7. Netfilter 网络过滤框架"){.xref}) 的优化，能够通过 MSS 来压缩包大小，路由器会用到 MMS 。参见 `iptables`{.literal}(8)中的\"TCPMSS\" .
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::
