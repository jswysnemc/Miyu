作为系统管理员，粗略地了解 Debian 系统的启动和配置方式是明智的。尽管准确的细节在安装的软件包及对应的文档中，但这些知识对我们大多数人来说都是必须掌握的。

下面是 Debian 系统初始化的要点概述。由于 Debian 系统在不断发展，您应该参考最新的文档。

::: itemizedlist
-   [Debian Linux 内核手册](https://kernel-team.pages.debian.net/kernel-handbook/index.html){.ulink} 是关于 Debian 内核的主要信息来源。

-   `bootup`{.literal}(7) 介绍了基于 `systemd`{.literal} 的系统启动流程。（近期的 Debian）

-   `boot`{.literal}(7) 介绍了基于 UNIX System V Release 4 的系统启动流程。（旧版的 Debian）
:::

## []{#_an_overview_of_the_boot_strap_process}3.1. 启动过程概述 {#启动过程概述 .title}

</div>

</div>
:::::

计算机系统从上电事件到能为用户提供完整的操作系统（OS）功能为止，需要经历几个阶段的[启动过程](https://zh.wikipedia.org/wiki/Booting){.ulink}。

为简便起见，笔者将讨论范围限定在具有默认安装的典型 PC 平台上。

典型的启动过程像是一个四级的火箭。每一级火箭将系统控制权交给下一级。

::: itemizedlist
-   [第 3.1.1 节 "第一阶段：UEFI"](ch03.zh-cn.html#_stage_1_the_uefi "3.1.1. 第一阶段：UEFI"){.xref}

-   [第 3.1.2 节 "第二阶段：引载加载程序"](ch03.zh-cn.html#_stage_2_the_boot_loader "3.1.2. 第二阶段：引载加载程序"){.xref}

-   [第 3.1.3 节 "第三阶段：迷你 Debian 系统"](ch03.zh-cn.html#_stage_3_the_mini_debian_system "3.1.3. 第三阶段：迷你 Debian 系统"){.xref}

-   [第 3.1.4 节 "第四阶段：常规 Debian 系统"](ch03.zh-cn.html#_stage_4_the_normal_debian_system "3.1.4. 第四阶段：常规 Debian 系统"){.xref}
:::

当然，这些阶段可以有不同的配置。比如，你编译了自己的内核，则可能会跳过迷你 Debian 系统的步骤。因此，在读者亲自确认之前，请勿假定自己系统的情况也是如此。

### []{#_stage_1_the_uefi}3.1.1. 第一阶段：UEFI {#第一阶段uefi .title}

</div>

</div>
:::::

[Unified Extensible Firmware Interface (UEFI) 统一可扩展固件接口](https://zh.wikipedia.org/wiki/Unified_Extensible_Firmware_Interface){.ulink} 定义了启动管理器作为 UEFI 规范的一部分。当一个计算机打开电源，启动管理器是启动流程的第一阶段，它检查启动配置并基于启动配置的设置，执行特定的操作系统引导加载程序或操作系统内核（通常是引导加载程序）。启动配置通过变量存储在 NVRAM，变量包括指示操作系统引导加载程序或操作系统内核的文件系统路径的变量。

[EFI system partition (ESP) EFI 系统分区](https://zh.wikipedia.org/wiki/EFI_system_partition){.ulink} 是一个数据存储设备分区，在计算机里用来遵照 UEFI 规范。当计算机打开电源时，由 UEFI 固件来访问，它存储了 UEFI 应用程序和这些应用程序运行所需要的文件，包括操作系统的引导加载程序。（在老的 PC 系统，存放在 [MBR](https://zh.wikipedia.org/wiki/Master_boot_record){.ulink} 里的 [BIOS](https://zh.wikipedia.org/wiki/BIOS){.ulink} 可以用来代替。）

### []{#_stage_2_the_boot_loader}3.1.2. 第二阶段：引载加载程序 {#第二阶段引载加载程序 .title}

</div>

</div>
:::::

[引导加载程序](https://zh.wikipedia.org/wiki/Boot_loader){.ulink}是启动过程的第二阶段，由 UEFI 启动。引导加载程序将系统内核映像和 [initrd](https://zh.wikipedia.org/wiki/Initrd){.ulink} 映像加载到内存并将控制权交给它们。initrd 映像是根文件系统映像，其支持程度依赖于所使用的引导加载程序。

Debian 系统通常使用 Linux 内核作为默认的系统内核。当前的 5.x Linux 内核的 initrd 映像在技术上是 [initramfs](https://wiki.debian.org/initramfs){.ulink}（初始 RAM 文件系统）映像。

对于 UEFI 系统，GRUB2 首先读取 ESP 分区，使用 \"`/boot/efi/EFI/debian/grub.cfg`{.literal}\"里面 `search.fs_uuid`{.literal} 指定的 UUID 来确定 GRUB2 菜单配置文件 \"`/boot/grub/grub.cfg`{.literal}\" 所在的分区。

GRUB2 菜单配置文件的关键部分看起来像：

``` screen
menuentry 'Debian GNU/Linux' ... {
        load_video
        insmod gzio
        insmod part_gpt
        insmod ext2
        search --no-floppy --fs-uuid --set=root fe3e1db5-6454-46d6-a14c-071208ebe4b1
        echo    'Loading Linux 5.10.0-6-amd64 ...'
        linux   /boot/vmlinuz-5.10.0-6-amd64 root=UUID=fe3e1db5-6454-46d6-a14c-071208ebe4b1 ro quiet
        echo    'Loading initial ramdisk ...'
        initrd  /boot/initrd.img-5.10.0-6-amd64
}
```

对于这部分的 `/boot/grub/grub.cfg`{.literal}，这个菜单条目的意义如下。

**表 3.2. `/boot/grub/grub.cfg`{.literal} 文件上面部分菜单条目意义**

::: table-contents
  设置                                  值
  ------------------------------------- -------------------------------------------------------------------------
  GRUB2 模块加载                        `gzio`{.literal}, `part_gpt`{.literal}, `ext2`{.literal}
  使用的根文件系统分区                  由 `UUID=fe3e1db5-6454-46d6-a14c-071208ebe4b1`{.literal} 指定的分区标识

:::

在 Debian 系统 ，\"`/boot/grub/grub.cfg`{.literal}\" 是由安装的 GRUB 软件包(如： `grub-efi-amd64`{.literal})管理，不赞成用户直接修改这个文件。应当定制 \"`/etc/grub.d/`{.literal}\" 和 \"`/etc/default/grub`{.literal}\" 来代替修改配置文件。配置 GRUB 配置文件，更新 NVRAM 变量来自动启动到 Debian：

``` screen
# dpkg-reconfigure grub-efi-amd64
```

### []{#_stage_3_the_mini_debian_system}3.1.3. 第三阶段：迷你 Debian 系统 {#第三阶段迷你-debian-系统 .title}

</div>

</div>
:::::

迷你 Debian 系统是启动流程的第三阶段，由引导加载程序启动。它会在内存中运行系统内核和根文件系统。这是启动流程的一个可选准备阶段。

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 "迷你 Debian 系统"是笔者自创的术语，用于在本文档中描述启动流程的第三个阶段。这个系统通常被称为 [initrd](https://zh.wikipedia.org/wiki/Initrd){.ulink} 或 initramfs 系统。内存中类似的系统在 [Debian 安装程序](https://www.debian.org/devel/debian-installer/){.ulink}中使用。
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

`/init`{.literal} 程序是内存中的根文件系统上执行的第一个程序。这个程序在用户空间把内核初始化，并把控制权交给下一阶段。迷你 Debian 系统能够在主引导流程之前添加内核模块或以加密形式挂载根文件系统，使引导流程更加灵活。

::: itemizedlist
-   如果 initramfs 是由 `initramfs-tools`{.literal} 创建，则\"`/init`{.literal}\" 程序是一个 shell 脚本程序。

    ::: itemizedlist
    -   通过给内核添加 "`break=init`{.literal}\" 等启动参数，你可以中断这部分启动流程以获取 root shell。更多中断条件请参见 "`/init`{.literal}" 脚本。这个 shell 环境已足够成熟，你可通过它很好地检查机器的硬件。

    -   迷你 Debian 系统中可用的命令是精简过的，且主要由一个称为 `busybox`{.literal}(1) 的 GNU 工具提供。
    :::

-   如果 initramfs 是由 `dracut`{.literal} 创建，则 \"`/init`{.literal}\" 程序是一个二进制 `systemd`{.literal} 程序。

    ::: itemizedlist
    -   迷你 Debian 系统中可用的命令是一个精简过的 `systemd`{.literal}(1) 环境。
    :::
:::

::: {.caution style="margin-left: 0.5in; margin-right: 0.5in;"}

                                    当在一个只读的根文件系统上时，使用 `mount`{.literal} 命令需要添加 `-n`{.literal} 选项。

### []{#_stage_4_the_normal_debian_system}3.1.4. 第四阶段：常规 Debian 系统 {#第四阶段常规-debian-系统 .title}

</div>

</div>
:::::

常规 Debian 系统是启动流程的第四阶段，由迷你 Debian 系统启动。迷你 Debian 系统的内核在此环境下继续运行。根文件系统将由内存切换到物理存储设备上。

[init](https://zh.wikipedia.org/wiki/Init){.ulink} 程序是系统执行的第一个程序（PID=1），它启动其它各种程序以完成主引导流程。init 程序的默认路径是 "`/usr/sbin/init`{.literal}"，但可通过内核启动参数修改，例如 "`init=/path/to/init_program`{.literal}\"。

在 Debian 8 jessie（2015 年发布）版本后，\"`/usr/sbin/init`{.literal}\" 是一个到 \"`/lib/systemd/systemd`{.literal}\" 的符号链接。

### []{#_systemd_init}3.3.1. Systemd 初始化 {#systemd-初始化 .title}

</div>

</div>
:::::

当 Debian 系统启动，`/usr/sbin/init`{.literal} 符号链接到的 `/usr/lib/systemd`{.literal} 作为初始系统进程 (`PID=1`{.literal}) 启动，该进程由 root (`UID=0`{.literal})所有。参见 `systemd`{.literal}(1)。

`systemd`{.literal} 初始化进程基于单元配置文件 (参见 `systemd.unit`{.literal}(5)) 来并行派生进程，这些单元配置文件使用声明样式来书写，代替之前的类 SysV 的过程样式。这些单元配置文件从下面的一系列路径来加载 (参见 `systemd-system.conf`{.literal}(5)) 。

派生的进程被放在一个单独的 [Linux control groups](https://www.kernel.org/doc/html/latest/admin-guide/cgroup-v2.html){.ulink}，在单元后命名，它们属于一个私有的 systemd 层级结构(参见 [cgroups](https://zh.wikipedia.org/wiki/Cgroups){.ulink} 和 [第 4.7.5 节 "Linux 安全特性"](ch04.zh-cn.html#_linux_security_features "4.7.5. Linux 安全特性"){.xref})。

系统模式单元从 `systemd.unit`{.literal}(5)描述中的\"System Unit Search Path\" 加载。主要部分是按照下列优先权顺序：

::: itemizedlist
-   \"`/etc/systemd/system/*`{.literal}\": 管理员创建的系统单元文件

-   \"`/run/systemd/system/*`{.literal}\": 运行时单元文件

-   \"`/lib/systemd/system/*`{.literal}\": 发行版软件包管理器安装的系统单元文件
:::

他们的相互依赖关系通过\"`Wants=`{.literal}\", \"`Requires=`{.literal}\", \"`Before=`{.literal}\", \"`After=`{.literal}\", ... 等指示来配置，(参见 `systemd.unit`{.literal}(5) 里的 \"MAPPING OF UNIT PROPERTIES TO THEIR INVERSES\")。 资源控制也是被定义 (参见 `systemd.resource-control`{.literal}(5)).

根据单元配置文件的后缀来区分它们的类型：

::: itemizedlist
-   [**\*.service**]{.strong} 描述由 `systemd`{.literal} 控制和监管的进程.参见 `systemd.service`{.literal}(5).

-   [**\*.device**]{.strong} 描述在 `sysfs`{.literal}(5) 里面作为 `udev`{.literal}(7) 设备树展示的设备。参见 `systemd.device`{.literal}(5).

-   [**\*.mount**]{.strong} 描述由 `systemd`{.literal} 控制和监管的文件系统挂载点。参见 `systemd.mount`{.literal}(5).

-   [**\*.automount**]{.strong} 描述由 `systemd`{.literal} 控制和监管的文件系统自动挂载点。参见 `systemd.automount`{.literal}(5).

-   [**\*.swap**]{.strong} 描述由 `systemd`{.literal} 控制和监管的 swap 文件或设备。参见 `systemd.swap`{.literal}(5).

-   [**\*.path**]{.strong} 描述被 `systemd`{.literal} 监控的路径，用于基于路径的活动。参见 `systemd.path`{.literal}(5).

-   [**\*.socket**]{.strong} 描述被 `systemd`{.literal} 控制和监管的套接字，用于基于套接字的活动。参见 `systemd.socket`{.literal}(5).

-   [**\*.timer**]{.strong} 描述被 `systemd`{.literal} 控制和监管的计时器，用于基于时间的活动。参见 `systemd.timer`{.literal}(5).

-   [**\*.slice**]{.strong} 管理 `cgroups`{.literal}(7) 的资源。参见 `systemd.slice`{.literal}(5).

-   [**\*.scope**]{.strong} 使用 `systemd`{.literal} 的总线接口来程序化的创建，用以管理一系列系统进程。 参见 `systemd.scope`{.literal}(5).

-   [**\*.target**]{.strong} 把其它单元配置文件分组，在启动的时候，来创建同步点。参见`systemd.target`{.literal}(5).
:::

系统启动时（即，init），`systemd`{.literal} 进程会尝试启动\"`/lib/systemd/system/default.target`{.literal}（通常是到\"`graphical.target`{.literal}\"的符号链接）。首先，一些特殊的 target 单元（参见 `systemd.special`{.literal}(7)），比如 \"`local-fs.target`{.literal}\"、\"`swap.target`{.literal}\"和\"`cryptsetup.target`{.literal}\"会被引入以挂载文件系统。之后，其它 target 单元也会根据单元依赖关系而被引入。详细情况，请阅读 `bootup`{.literal}(7)。

`systemd`{.literal} 提供向后兼容的功能。在 \"`/etc/init.d/rc[0123456S].d/[KS]`{.literal}[*`name`{.literal}*]{.emphasis}\" 里面的 SysV 风格的启动脚本仍然会被分析；`telinit`{.literal}(8) 会被转换为 systemd 的单元活动请求。

## []{#_the_system_message}3.5. 系统消息 {#系统消息 .title}

</div>

</div>
:::::

在 `systemd`{.literal} 下, 内核和系统的信息都通过日志服务 `systemd-journald.service`{.literal} (又名 `journald`{.literal})来记录，放在\"`/var/log/journal`{.literal}\"下的不变的二进制数据，或放在\"`/run/log/journal/`{.literal}\"下的变化的二进制数据.这些二进制日志数据，可以通过 `journalctl`{.literal}(1) 命令来访问。例如，你可以显示从最后一次启动以来的日志，按如下所示：

``` screen
$ journalctl -b
```

  查看从最后一次启动开始的 \"`$unit`{.literal}\"的工作日志 (\"`tail -f`{.literal}\" 式样)   \"`journalctl -b -u $unit -f`{.literal}\"

### []{#_customization_example_to_tweak_sshd_service}3.8.5. 调整 sshd 服务的个性化例子 {#调整-sshd-服务的个性化例子 .title}

</div>

</div>
:::::

使用默认安装，通过 `systemd`{.literal} 启动的过程中，在 `network.target`{.literal} 启动后，很多网络服务 (参见 [第 6 章 *网络应用*](ch06.zh-cn.html "第 6 章 网络应用"){.xref})作为后台守护进程（daemon）启动。 \"`sshd`{.literal}\" 也不列外。让我们修改为按需启动\"`sshd`{.literal}\" 作为一个定制化的例子。

首先，禁用系统安装的服务单元。

``` screen
 $ sudo systemctl stop sshd.service
 $ sudo systemctl mask sshd.service
```

传统 Unix 服务的按需套接字激活（on-demand socket activation）系统由 `inetd`{.literal} (或 `xinetd`{.literal})超级服务来提供。在 `systemd`{.literal} 下, 相同功能能够通过增加[**\*.socket**]{.strong} 和 [**\*.service**]{.strong} 单元配置文件来启用。

`sshd.socket`{.literal} 用来定义一个监听的套接字

``` screen
[Unit]
Description=SSH Socket for Per-Connection Servers

[Socket]
ListenStream=22
Accept=yes

[Install]
WantedBy=sockets.target
```

`sshd@.service`{.literal} 作为 `sshd.socket`{.literal} 匹配的服务文件

``` screen
[Unit]
Description=SSH Per-Connection Server

[Service]
ExecStart=-/usr/sbin/sshd -i
StandardInput=socket
```

然后重新加载。

``` screen
 $ sudo systemctl daemon-reload
```

## []{#_the_kernel_module_initialization}3.10. 内核模块初始化 {#内核模块初始化 .title}

</div>

</div>
:::::

通过 `modprobe`{.literal}(8) 程序添加和删除内核模块，使我们能够从用户进程来配置正在运行的 Linux 内核。udev 系统(参见 [第 3.9 节 "udev 系统"](ch03.zh-cn.html#_the_udev_system "3.9. udev 系统"){.xref})自动化它的调用来帮助内核模块初始化。

下面的非硬件模块和特殊的硬件驱动模块，需要被预先加载，把它们在\"`/etc/modules`{.literal}\"文件里列出 (参见 `modules`{.literal}(5)).

::: itemizedlist
-   [TUN/TAP](https://zh.wikipedia.org/wiki/TUN/TAP){.ulink} 模块提供虚拟的 Point-to-Point 网络设备 (TUN) 和虚拟的 Ethernet 以太网网络设备 (TAP),

-   [netfilter](https://zh.wikipedia.org/wiki/Netfilter){.ulink} 模块提供 netfilter 防火墙能力(`iptables`{.literal}(8), [第 5.7 节 "Netfilter 网络过滤框架"](ch05.zh-cn.html#_netfilter_infrastructure "5.7. Netfilter 网络过滤框架"){.xref}),

-   [watchdog timer](https://zh.wikipedia.org/wiki/Watchdog_timer){.ulink} 驱动模块。
:::

`modprobe`{.literal}(8) 程序的配置文件是按 `modprobe.conf`{.literal}(5)的说明放在\"`/etc/modprobes.d/`{.literal}\" 目录下，(如果你想避免自动加载某些内核模块，考虑把它们作为黑名单放在\"`/etc/modprobes.d/blacklist`{.literal}\" 文件里.)

\"`/lib/modules/`{.literal}[*`version`{.literal}*]{.emphasis}`/modules.dep`{.literal}\" 文件由 `depmod`{.literal}(8) 程序生成，它描述了 `modprobe`{.literal}(8) 程序使用的模块依赖性.

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 如果你在启动时出现模块加载问题，或者 `modprobe`{.literal}(8)时出现模块加载问题, \"`depmod -a`{.literal}\" 可以通过重构\"`modules.dep`{.literal}\"来解决这些问题。
  ------------------------------ -------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

`modinfo`{.literal}(8) 程序显示 Linux 内核模块信息。

`lsmod`{.literal}(8) 程序以好看的格式展示\"`/proc/modules`{.literal}\"的内容,显示当前内核加载了哪些模块。

|                                   |                                                                                                                                              |
|                                   | 你可以在启动时配置硬件来激活期望的硬件特征。参见 [第 9.5.4 节 "硬件配置"](ch09.zh-cn.html#_hardware_configuration "9.5.4. 硬件配置"){.xref}. |
|                                   |                                                                                                                                              |
|                                   | 你可以重新编译内核来增加你的特殊设备的支持。参见 [第 9.10 节 "内核"](ch09.zh-cn.html#_the_kernel "9.10. 内核"){.xref}.                       |
