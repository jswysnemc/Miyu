## []{#_normal_unix_authentication}4.1. 一般的 Unix 认证 {#一般的-unix-认证 .title}

</div>

</div>
:::::

一般的 Unix 认证由 [PAM （Pluggable Authentication Modules，即可插入的验证模块）](https://zh.wikipedia.org/wiki/Pluggable_Authentication_Modules){.ulink} 下的 `pam_unix`{.literal}(8) 模块提供。它的 3 个重要文件如下，其内的条目使用 "`:`{.literal}" 分隔。

:::: table
[]{#dimportantconfigilesforpam_unixi}

**表 4.1. `pam_unix`{.literal}(8) 使用的 3 个重要配置文件**

::: table-contents
  文件                      权限                     用户               组                   说明
  ------------------------- ------------------------ ------------------ -------------------- ------------------------
  `/etc/passwd`{.literal}   `-rw-r--r--`{.literal}   `root`{.literal}   `root`{.literal}     （明文的）用户账号信息
  `/etc/shadow`{.literal}   `-rw-r-----`{.literal}   `root`{.literal}   `shadow`{.literal}   安全加密的用户账号信息
  `/etc/group`{.literal}    `-rw-r--r--`{.literal}   `root`{.literal}   `root`{.literal}     组信息
:::
::::

\

"`/etc/passwd`{.literal}" 包含下列内容。

``` screen
 ...
user1:x:1000:1000:User1 Name,,,:/home/user1:/bin/bash
user2:x:1001:1001:User2 Name,,,:/home/user2:/bin/bash
 ...
```

如 `passwd`{.literal}(5) 中所述，这个文件中被 "`:`{.literal}" 分隔的每项含义如下。

::: itemizedlist
-   登录名

-   密码形式说明

-   数字形式的用户 ID

-   数字形式的组 ID

-   用户名或注释字段

-   用户家目录

-   可选的用户命令解释器
:::

"`/etc/passwd`{.literal}" 的第二项曾经被用来保存加密后的密码。在引入了 "`/etc/shadow`{.literal}" 后，该项被用来说明密码形式。

:::: table
[]{#thesecondentrycontentofetcpasswd}

**表 4.2. "`/etc/passwd`{.literal}" 第二项的内容**

::: table-contents
  内容     说明
  -------- ----------------------------------------------
  （空）   无需密码的账号
  x        加密后的密码保存在 "`/etc/shadow`{.literal}"
:::
::::

\

"`/etc/shadow`{.literal}" 包含下列内容。

``` screen
 ...
user1:$1$Xop0FYH9$IfxyQwBe9b8tiyIkt2P4F/:13262:0:99999:7:::
user2:$1$vXGZLVbS$ElyErNf/agUDsm1DehJMS/:13261:0:99999:7:::
 ...
```

如 `shadow`{.literal}(5) 中所述，这个文件中被 "`:`{.literal}" 分隔的每项含义如下。

::: itemizedlist
-   登录名

-   加密后的密码（开头的 "`$1$`{.literal}" 表示使用 MD5 加密。"\*" 表示无法登录。）

-   最后一次修改密码的时间，其表示从 1970 年 1 月 1 日起的天数

-   允许用户再次修改密码的天数间隔

-   用户必须修改密码的天数间隔

-   密码失效前的天数，在此期间用户会被警告

-   密码失效后的天数，在次期间密码依旧会被接受

-   账号失效的时间，其表示从 1970 年 1 月 1 日起的天数

-   ...
:::

"`/etc/group`{.literal}" 包含下列内容。

``` screen
group1:x:20:user1,user2
```

如 `group`{.literal}(5) 中所述，这个文件中被 "`:`{.literal}" 分隔的每项含义如下。

::: itemizedlist
-   组名称

-   加密后的密码（不会被真正使用）

-   数字形式的组 ID

-   使用 "," 分隔的用户名列表
:::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ ----------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 "`/etc/gshadow`{.literal}" 为 "`/etc/group`{.literal}" 提供了与 "`/etc/shadow`{.literal}" 相似的功能，但没有被真正地使用。
  ------------------------------ ----------------------------------------------------------------------------------------------------------------------------
:::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 如果\"`auth optional pam_group.so`{.literal}\" 这行添加到了\"`/etc/pam.d/common-auth`{.literal}\"，并且在\"`/etc/security/group.conf`{.literal}\" 里进行了设置，一个用户的实际组就可以被动态添加。参见 `pam_group`{.literal}(8).
  ------------------------------ ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 `base-passwd`{.literal} 软件包包含了一份用户和组的官方文档："`/usr/share/doc/base-passwd/users-and-groups.html`{.literal}"。
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------

## []{#_good_password}4.3. 好密码 {#好密码 .title}

</div>

</div>
:::::

在系统安装时建立一个账号或使用 `passwd`{.literal}(1) 命令时，你应该选择一个[好密码](https://zh.wikipedia.org/wiki/Password_strength){.ulink}，它应该由 6 到 8 个字符组成，其中包含下列根据 `passwd`{.literal}(1) 设定的每个组合中的一个或多个字符。

::: itemizedlist
-   小写字母

-   数字 0 到 9

-   标点符号
:::

::: {.warning style="margin-left: 0.5in; margin-right: 0.5in;"}
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[警告\]](images/warning.png)  警告
                                    密码中不要使用可以猜到的词。账号名、身份证号码、电话号码、地址、生日、家庭成员或宠物的名字、字典单词、简单的字符序列（例如 "12345" 或 "qwerty"）等都是糟糕的选择。
  --------------------------------- --------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

## []{#_pam_and_nss}4.5. PAM 和 NSS {#pam-和-nss .title}

</div>

</div>
:::::

现代的[类 Unix](https://zh.wikipedia.org/wiki/Unix-like){.ulink} 系统（例如 Debian 系统）提供 [PAM （Pluggable Authentication Modules，插入式验证模块）](https://zh.wikipedia.org/wiki/Pluggable_Authentication_Modules){.ulink} 和 [NSS（Name Service Switch，名称服务切换）](https://zh.wikipedia.org/wiki/Name_Service_Switch){.ulink} 机制给本地系统管理员，使他们能够配置自己的系统。它们的功能可以概括为以下几点。

::: itemizedlist
-   PAM 给应用软件提供了一个灵活的认证机制，因此涉及到了密码数据的交换。

-   NSS 提供了一个灵活的名称服务机制，它经常被 [C 标准库](https://zh.wikipedia.org/wiki/C_standard_library){.ulink}使用，使例如 `ls`{.literal}(1) 和 `id`{.literal}(1) 这样的程序获得用户和组名称。
:::

PAM 和 NSS 系统必须保持配置一致。

### []{#_the_modern_centralized_system_management}4.5.2. 现代的集中式系统管理 {#现代的集中式系统管理 .title}

</div>

</div>
:::::

现代的集中式系统管理可以使用集中式的[轻量目录访问协议（LDAP）](https://zh.wikipedia.org/wiki/Lightweight_Directory_Access_Protocol){.ulink}服务器进行部署，从而通过网络管理许多类 Unix 和 非类 Unix 系统。轻量目录访问协议的开源实现是 [OpenLDAP 软件](https://www.openldap.org/){.ulink}。

LDAP 服务器使用带有 PAM 和 NSS 的`libpam-ldapd`{.literal} 和 `libnss-ldapd`{.literal} 软件包为 Debian 系统提供账号信息。需要一些动作来启用 LDAP（我没有使用过这个设置，并且下面的信息纯粹是第二手的信息。请在这种前提下阅读下列内容。）。

::: itemizedlist
-   你通过运行一个程序，例如独立的 LDAP 守护进程 `slapd`{.literal}(8)，来建立集中式的 LDAP 服务器。

-   你在 "`/etc/pam.d/`{.literal}" 目录中的 PAM 配置文件里，使用 "`pam_ldap.so`{.literal}" 替代默认值 "`pam_unix.so`{.literal}"。

-   你在 "`/etc/nsswitch.conf`{.literal}" 文件中改变 NSS 配置，使用 "`ldap`{.literal}" 替代默认值（"`compat`{.literal}" 或 "`file`{.literal}"）。

-   为了密码的安全，你必须让 `libpam-ldapd`{.literal} 使用 [SLL（或 TLS）](https://zh.wikipedia.org/wiki/Transport_Layer_Security){.ulink}连接。

-   为了确保 LDAP 网络开销数据的完整性，你必须让 `libpam-ldapd`{.literal} 使用 [SLL（或 TLS）](https://zh.wikipedia.org/wiki/Transport_Layer_Security){.ulink}连接。

-   为了减少 LDAP 网络流量，你应该在本地运行 `nscd`{.literal}(8) 来缓存任何 LDAP 搜索结果。
:::

参见由 `libpam-doc`{.literal} 软件包提供的 `nsswitch.conf`{.literal}(5)、`pam.conf`{.literal}(5)、 `ldap.conf`{.literal}(5) 和 "`/usr/share/doc/libpam-doc/html/`{.literal}" 中的文档，以及 `glibc-doc`{.literal} 软件包提供的 "`info libc 'Name Service Switch'`{.literal}"。

类似地，你可以使用其它方法来设置另一种集中式的系统。

::: itemizedlist
-   同 Windows 系统集成用户和组。

    ::: itemizedlist
    -   通过 `winbind`{.literal} 和 `libpam_winbind`{.literal} 软件包访问 [Windows domain](https://zh.wikipedia.org/wiki/Windows_domain){.ulink} 服务.

    -   参见 `winbindd`{.literal}(8) 和 [Integrating MS Windows Networks with Samba](https://www.samba.org/samba/docs/man/Samba-HOWTO-Collection/integrate-ms-networks.html){.ulink}。
    :::

-   同古老的类 Unix 系统集成用户和组。

    ::: itemizedlist
    -   通过 `nis`{.literal} 软件包访问 [NIS (之前叫 YP)](https://zh.wikipedia.org/wiki/Network_Information_Service){.ulink} 或 [NIS+](https://zh.wikipedia.org/wiki/NIS+){.ulink}。

    -   参见 [The Linux NIS(YP)/NYS/NIS+ HOWTO](https://tldp.org/HOWTO/NIS-HOWTO/){.ulink}。
    :::

### []{#_secure_password_on_the_internet}4.6.1. 确保互联网上的的密码安全 {#确保互联网上的的密码安全 .title}

</div>

</div>

许多流行的传输层服务都使用纯文本来传输包括密码验证信息在内的各类消息。使用纯文本在公网上传输密码是很糟糕的做法，因为这样传输的密码很容易在网上被他人截获。为了确保整个沟通过程，包括密码信息在内都使用加密传输来确保安全，您可以在"[传输层安全（Transport Layer Security，TLS）](https://zh.wikipedia.org/wiki/Transport_Layer_Security){.ulink}"协议或者其前身，"安全套接字层（Secure Sockets Layer，SSL）"协议之上运行这些服务。

### []{#_securing_the_root_password}4.6.4. root 密码安全 {#root-密码安全 .title}

</div>

</div>
:::::

为阻止人们使用 root 权限访问你的机器，你需要做下面的操作。

::: itemizedlist
-   防止对系统存储设备的物理访问 ([HDD](https://zh.wikipedia.org/wiki/Hard_disk_drive){.ulink} / [SSD](https://zh.wikipedia.org/wiki/Solid-state_drive){.ulink} / \...)

-   锁住 UEFI/ BIOS 来阻止从可移动介质启动

-   为 GRUB 交互式会话设置密码

-   锁住 GRUB 菜单，禁止编辑
:::

如果可以物理访问系统存储设备，则可以使用下面的步骤，相对简单的重置密码。

::: orderedlist
1.  将系统存储设备拿到一个可以设置 UEFI/BIOS 从 USB 启动的电脑。

2.  使用应急救援介质启动系统 (参见 [第 3.2.2 节 "USB 上的 Linux 即用救援系统"](ch03.zh-cn.html#_linux_live_rescue_system_on_usb "3.2.2. USB 上的 Linux 即用救援系统"){.xref})。

3.  用读写访问挂载根分区。

4.  编辑根分区的\"`/etc/passwd`{.literal}\"文件，使 `root`{.literal} 账户条目的第二段为空。
:::

对于 `grub-rescue-pc`{.literal} ，即使用紧急介质启动的电脑，如果有编辑 GRUB 菜单条目 (参见 [第 3.1.2 节 "第二阶段：引载加载程序"](ch03.zh-cn.html#_stage_2_the_boot_loader "3.1.2. 第二阶段：引载加载程序"){.xref}) 的权限，在启动时，使用下面的步骤更加简单。

::: orderedlist
1.  使用内核参数启动系统来修改一些事情，比如说，\"`root=/dev/sda6 rw init=/bin/sh`{.literal}\".

2.  编辑 \"`/etc/passwd`{.literal}\" 文件，使 `root`{.literal} 账户条目的第二段为空。

3.  重启系统。
:::

系统的 root shell 现在可以无密码访问了。

::: {.note style="margin-left: 0.5in; margin-right: 0.5in;"}
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
   ![\[注意\]](images/note.png)  注意
                                 一旦某人拥有 root shell 访问权限，他能够访问任何内容，并可以重设系统上的任何密码。此外，他可以使用 `john`{.literal} 和 `crack`{.literal} 等软件包的暴力破解工具来比较所有用户的密码 (参见 [第 9.5.11 节 "系统安全性和完整性检查"](ch09.zh-cn.html#_system_security_and_integrity_check "9.5.11. 系统安全性和完整性检查"){.xref})。被破解的密码，可以用来和其它系统进行比较。
  ------------------------------ ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------
:::

为避免这些相关问题，仅有的理论上的软件解决方案是使用 [dm-crypt](https://zh.wikipedia.org/wiki/Dm-crypt){.ulink} 和 initramfs (参见 [第 9.9 节 "数据加密提示"](ch09.zh-cn.html#_data_encryption_tips "9.9. 数据加密提示"){.xref})加密 root 分区(或 \"`/etc`{.literal}\" 分区) 。这样的话，你总是需要密码来启动系统。

### []{#_access_control_lists}4.7.1. 访问控制列表(ACLs) {#访问控制列表acls .title}

</div>

</div>
:::::

访问控制列表 ACL 是在 [第 1.2.3 节 "文件系统权限"](ch01.zh-cn.html#_filesystem_permissions "1.2.3. 文件系统权限"){.xref} 里面解释的普通权限的一个超集。

在现代桌面环境中，你会遇到 ACL 行为。比如，当一个已经格式化的 USB 存储设备自动挂载到 \"`/media/penguin/USBSTICK`{.literal}\"，一个普通用户 `penguin`{.literal} 能够执行：

``` screen
 $ cd /media/penguin
 $ ls -la
total 16
drwxr-x---+ 1 root    root    16 Jan 17 22:55 .
drwxr-xr-x  1 root    root    28 Sep 17 19:03 ..
drwxr-xr-x  1 penguin penguin 18 Jan  6 07:05 USBSTICK
```

在第 11 列的 \"`+`{.literal}\" 表示 ACL 在使用。如果没有 ACL，一个普通用户 `penguin`{.literal} 应该不能够像这样列出目录内容，因为 `penguin`{.literal} 不在 `root`{.literal} 组。你能够按如下方式查看 ACL：

``` screen
 $ getfacl .
# file: .
# owner: root
# group: root
user::rwx
user:penguin:r-x
group::---
mask::r-x
other::---
```

这里：

::: itemizedlist
-   \"`user::rwx`{.literal}\"、\"`group::---`{.literal}\" 和 \"`other::---`{.literal}\" 相应的为普通所有者、组和其它人的权限。

-   ACL \"`user:penguin:r-x`{.literal}\" 运行普通用户 `penguin`{.literal} 有 \"`r-x`{.literal}\" 权限。这个能够让 \"`ls -la`{.literal}\" 列出目录内容。

-   ACL \"`mask::r-x`{.literal}\" 设置上层绑定权限。
:::

更多信息参见 \"[POSIX Access Control Lists on Linux](https://www.usenix.org/legacy/events/usenix03/tech/freenix03/full_papers/gruenbacher/gruenbacher.pdf){.ulink}\"、`acl`{.literal}(5)、 `getfacl`{.literal}(1) 和 `setfacl`{.literal}。

### []{#_restricting_access_to_some_server_services}4.7.4. 限制访问某些服务端的服务 {#限制访问某些服务端的服务 .title}

</div>

</div>
:::::

对系统安全而言，尽可能的禁用服务程序，是一个好的主意。网络服务是危险的。有不使用的服务，不管是直接由[后台守护进程（daemon）](https://zh.wikipedia.org/wiki/Daemon_(computer_software)){.ulink}激活，还是通过[super-server](https://zh.wikipedia.org/wiki/Super-server){.ulink} 程序激活，都被认为是安全风险。

许多程序,比如说 `sshd`{.literal}(8), 使用基于 PAM 的访问控制。也还有许多方式来限制访问一些服务端的程序。

::: itemizedlist
-   配置文件: \"`/etc/default/`{.literal}[*`program_name`{.literal}*]{.emphasis}\"

-   [后台守护进程（daemon）](https://zh.wikipedia.org/wiki/Daemon_(computer_software)){.ulink}的 Systemd 服务单元配置

-   [PAM (Pluggable Authentication Modules)](https://zh.wikipedia.org/wiki/Pluggable_Authentication_Modules){.ulink}

-   [super-server](https://zh.wikipedia.org/wiki/Super-server){.ulink} 使用 \"`/etc/inetd.conf`{.literal}\"

-   [TCP wrapper](https://zh.wikipedia.org/wiki/TCP_Wrapper){.ulink} 使用 \"`/etc/hosts.deny`{.literal}\" 和 \"`/etc/hosts.allow`{.literal}\" , `tcpd`{.literal}(8)

-   [Sun RPC](https://zh.wikipedia.org/wiki/Open_Network_Computing_Remote_Procedure_Call){.ulink}\" 使用 `/etc/rpc.conf`{.literal}\"

-   `atd`{.literal}(8) 使用 \"`/etc/at.allow`{.literal}\" 和 \"`/etc/at.deny`{.literal}\"

-   `crontab`{.literal}(1) 使用 \"`/etc/cron.allow`{.literal}\" 和 \"`/etc/cron.deny`{.literal}\"

-   [Network firewall](https://zh.wikipedia.org/wiki/Firewall){.ulink} 或[netfilter](https://zh.wikipedia.org/wiki/Netfilter){.ulink} 框架
:::

参见 [第 3.6 节 "系统管理"](ch03.zh-cn.html#_system_management_operations "3.6. 系统管理"){.xref}、[第 4.5.1 节 "PAM 和 NSS 访问的配置文件"](ch04.zh-cn.html#_configuration_files_accessed_by_pam_and_nss "4.5.1. PAM 和 NSS 访问的配置文件"){.xref} 和 [第 5.7 节 "Netfilter 网络过滤框架"](ch05.zh-cn.html#_netfilter_infrastructure "5.7. Netfilter 网络过滤框架"){.xref}。

::: {.tip style="margin-left: 0.5in; margin-right: 0.5in;"}

### []{#_linux_security_features}4.7.5. Linux 安全特性 {#linux-安全特性 .title}

</div>

</div>
:::::

Linux 内核已经发展和支持在传统的 UNIX 实现里面没有的安全特征。

Linux 支持 [扩展属性](https://zh.wikipedia.org/wiki/Extended_file_attributes){.ulink}，扩展了传统的 UNIX 属性 (参见 `xattr`{.literal}(7))。

Linux 把传统的超级用户相关的特权分开到不同的单元，被称为 `capabilities`{.literal}(7)，它能够独立的启用和禁用。从 2.2 版本内核开始，Capabilities 是一个线程独立的属性。

[Linux Security Module (LSM) 安全模块框架](https://zh.wikipedia.org/wiki/Linux_Security_Modules){.ulink} 提供了一个[多方面的安全检查机制](https://zh.wikipedia.org/wiki/Mandatory_access_control){.ulink}，和新的内核扩展关联。例如：

::: itemizedlist
-   [AppArmor](https://zh.wikipedia.org/wiki/AppArmor){.ulink}

-   [Security-Enhanced Linux (SELinux)](https://zh.wikipedia.org/wiki/Security-Enhanced_Linux){.ulink}

-   [Smack (Simplified Mandatory Access Control Kernel)](https://zh.wikipedia.org/wiki/Smack_(software)){.ulink}

-   [Tomoyo Linux](https://zh.wikipedia.org/wiki/Tomoyo_Linux){.ulink}
:::

这些扩展紧缩的权力模型比普通的类 Unix 安全模型策略更加严格，甚至 root 的权力也被限制。建议你阅读 [kernel.org 上的 Linux 安全模块（LSM）框架文档](https://www.kernel.org/doc/html/latest/admin-guide/LSM/index.html){.ulink}。

Linux 的 [namespaces](https://zh.wikipedia.org/wiki/Linux_namespaces){.ulink} 封装了一个全局系统资源到一个抽象的概念，全局系统资源在 namespace 内对进程可见，并且 namespace 有它们自己的全局资源隔离实例。 对其它进程全局资源的可见性的改变是，同一个 namespace 的成员可见，但是对非同一个 namespace 的其它进程不可见。从内核 5.6 版本起，有 8 种 namespaces (参见 `namespaces`{.literal}(7), `unshare`{.literal}(1), `nsenter`{.literal}(1))。

在 Debian 11 Bullseye (2021) 中, Debian 使用 unified cgroup hierarchy（统一 cgroup 层级架构） (亦称为 [cgroups-v2](https://www.kernel.org/doc/html/latest/admin-guide/cgroup-v2.html){.ulink})。

[namespaces](https://zh.wikipedia.org/wiki/Linux_namespaces){.ulink} 同 [cgroups](https://zh.wikipedia.org/wiki/Cgroups){.ulink} 一起来隔离它们的进程，允许资源控制的使用示例是：

::: itemizedlist
-   [Systemd](https://zh.wikipedia.org/wiki/Systemd){.ulink}。参见 [第 3.3.1 节 "Systemd 初始化"](ch03.zh-cn.html#_systemd_init "3.3.1. Systemd 初始化"){.xref}。

-   [沙盒环境](https://zh.wikipedia.org/wiki/Sandbox_(computer_security)){.ulink}。参见 [第 7.7 节 "沙盒"](ch07.zh-cn.html#_sandbox "7.7. 沙盒"){.xref}。

-   [Linux 容器](https://zh.wikipedia.org/wiki/List_of_Linux_containers){.ulink}，比如 [Docker](https://zh.wikipedia.org/wiki/Docker_(software)){.ulink}、 [LXC](https://zh.wikipedia.org/wiki/LXC){.ulink}。参见 [第 9.11 节 "虚拟化系统"](ch09.zh-cn.html#_virtualized_system "9.11. 虚拟化系统"){.xref}。
:::

这些功能不能够通过 [第 4.1 节 "一般的 Unix 认证"](ch04.zh-cn.html#_normal_unix_authentication "4.1. 一般的 Unix 认证"){.xref} 实现。这些高级话题大部分超出了本介绍文档的范围。
