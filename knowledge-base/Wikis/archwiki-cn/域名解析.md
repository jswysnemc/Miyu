通常，一个[域名](<https://en.wikipedia.org/wiki/Domain_name> "wikipedia:Domain name")代表了一个 IP 地址，并在[域名系统](<https://en.wikipedia.org/wiki/Domain_Name_System> "wikipedia:Domain Name System")（Domain Name System，DNS）中与其关联。本文将介绍如何配置域名解析和解析域名。 

##  名称服务转换（NSS）

[名称服务转换](<https://en.wikipedia.org/wiki/Name_Service_Switch> "wikipedia:Name Service Switch")（Name Service Switch，NSS）工具是 GNU C 库（[glibc](<https://archlinux.org/packages/?name=glibc>)包）的一部分，支持 [getaddrinfo(3)](<https://man.archlinux.org/man/getaddrinfo.3>) API，用于解析域名。NSS 允许系统数据库由单独的服务提供，其搜索顺序可由管理员在 [nsswitch.conf(5)](<https://man.archlinux.org/man/nsswitch.conf.5>) 中配置。负责响应域名解析的数据库是 _hosts_ 数据库，glibc 为此提供了下列服务： 

  * _files_ （文件）：读取 `/etc/hosts` 文件，参见 [hosts(5)](<https://man.archlinux.org/man/hosts.5>)
  * _dns_ ：读取 `/etc/resolv.conf` 的 [glibc 解析器](<#Glibc_%E8%A7%A3%E6%9E%90%E5%99%A8>)，参见 [resolv.conf(5)](<https://man.archlinux.org/man/resolv.conf.5>)

[systemd](<../zh-cn/Systemd.html> "Systemd") 为主机名解析提供了三项 NSS 服务： 

  * [nss-resolve(8)](<https://man.archlinux.org/man/nss-resolve.8>)——缓存 DNS 存根解析器，[systemd-resolved](<../zh-cn/Systemd-resolved.html> "Systemd-resolved") 一文中有描述
  * [nss-myhostname(8)](<https://man.archlinux.org/man/nss-myhostname.8>)——无需编辑 `/etc/hosts` 即可提供本地主机名解析，[网络配置#Local hostname resolution](<../zh-cn/%E7%BD%91%E7%BB%9C%E9%85%8D%E7%BD%AE.html#Local_hostname_resolution> "网络配置") 一文中有描述
  * [nss-mymachines(8)](<https://man.archlinux.org/man/nss-mymachines.8>)——为本地 [systemd-machined(8)](<https://man.archlinux.org/man/systemd-machined.8>) 容器的名称提供本地主机名解析

###  使用 NSS 解析域名

NSS 数据库可使用 [getent(1)](<https://man.archlinux.org/man/getent.1>) 查询。可以使用以下命令通过 NSS 解析域名： 
    
    $ getent ahosts _domain_name（域名）_
    
**注意：** 虽然大多数程序使用 NSS 解析域名，但是有些程序可能会直接读取 `/etc/resolv.conf` 和/或 `/etc/hosts`。参见 [网络配置#局域网主机名解析](<../zh-cn/%E7%BD%91%E7%BB%9C%E9%85%8D%E7%BD%AE.html#%E5%B1%80%E5%9F%9F%E7%BD%91%E4%B8%BB%E6%9C%BA%E5%90%8D%E8%A7%A3%E6%9E%90> "网络配置")。

  * [Netctl#/etc/resolv.conf](<../zh-cn/Netctl.html#/etc/resolv.conf> "Netctl")

要防止程序覆盖 `/etc/resolv.conf`，还可以通过设置不可变[文件属性](</wzh/index.php?title=File_attribute&action=edit&redlink=1> "File attribute（页面不存在）")来为其建立写保护：

**提示：** 如果想要多个进程写入 `/etc/resolv.conf`，可以使用 [resolvconf](</wzh/index.php?title=Resolvconf&action=edit&redlink=1> "Resolvconf（页面不存在）")。

####  使用 nmcli 的替代方案

[![](../File:Merge-arrows-2.png)](<../File:Merge-arrows-2.png>)**本文或本章节可能需要合并到[NetworkManager#/etc/resolv.conf](<../zh-cn/NetworkManager.html#/etc/resolv.conf> "NetworkManager")。**

如果使用 [NetworkManager](<../zh-cn/NetworkManager.html> "NetworkManager")，可以使用 [nmcli(1)](<https://man.archlinux.org/man/nmcli.1>) 为 `/etc/resolv.conf` 设置持久选项。将下面的 _Wired_ 改为连接名称。例如

有关更多选项，请参阅手册 [nmcli(1)](<https://man.archlinux.org/man/nmcli.1>)、[nm-settings-nmcli(5)](<https://man.archlinux.org/man/nm-settings-nmcli.5>) 和 [resolv.conf(5)](<https://man.archlinux.org/man/resolv.conf.5>)。 

###  限制查找时间

如果您遇到了非常长时间查找主机名的情况（可能是在 [pacman](<../zh-cn/Pacman.html> "Pacman") 里或者在浏览网页时），定义一个小的超时时间通常能对问题有所帮助，在此之后系统会使用备用域名服务器（nameserver）。为此，请在 `/etc/resolv.conf` 中添加以下内容。 
    
    /etc/resolv.conf
    
    options timeout:1

###  IPv6 主机名查找延迟

如果在解析主机名时遇到 5 秒延迟的情况，原因可能是 DNS 服务器/防火墙行为不当，只对并行的 A 和 AAAA 请求给出了一个回应。[[1]](<https://udrepper.livejournal.com/20948.html>) 可以在 `/etc/resolv.conf` 中设置以下选项来修复问题： 
    
    /etc/resolv.conf
    
    options single-request

###  本地域名

为了能够使用本地机器名称的主机名，而无需使用格式完全合格的域名，请在 `/etc/resolv.conf` 中添加一行本地域名，如： 
    
    /etc/resolv.conf
    
    search _example.org_

这样当使用 _ssh_ 命令时，可以把 `mainmachine1.example.org` 这样的本地主机简单地称为 `mainmachine1`，不过使用 [drill](<#%E6%9F%A5%E6%89%BE%E5%AE%9E%E7%94%A8%E5%B7%A5%E5%85%B7>) 命令时仍需要格式完全合格的域名以便查找。 

##  查找实用工具

要查询特定的 DNS 服务器和 DNS/[DNSSEC](</wzh/index.php?title=DNSSEC&action=edit&redlink=1> "DNSSEC（页面不存在）") 记录，可以使用专用 DNS 查询实用工具。这些工具自身实现了 DNS 功能，不使用 [NSS](<#%E5%90%8D%E7%A7%B0%E6%9C%8D%E5%8A%A1%E8%BD%AC%E6%8D%A2%EF%BC%88NSS%EF%BC%89>)。 

[ldns](<https://archlinux.org/packages/?name=ldns>)包 软件包提供了 [drill(1)](<https://man.archlinux.org/man/drill.1>) 命令，是一个旨在检索 DNS 信息的工具。 

例如，要用 _drill_ 查询特定名称服务器（nameserver）获取一个域名的 TXT 记录，可执行： 
    
    $ drill @_nameserver_ TXT _domain（域名）_
    
除非指定了一个 DNS 服务器，否则 _drill_ 将使用 `/etc/resolv.conf` 中定义的名称服务器。 

**提示：** 一些 DNS 服务器附带它们自己的 DNS 查找实用工具。例如： 

  * [knot](<https://archlinux.org/packages/?name=knot>)包 软件包提供了 [khost(1)](<https://man.archlinux.org/man/khost.1>) 和 [kdig(1)](<https://man.archlinux.org/man/kdig.1>) 命令。
  * [Unbound](<../zh-cn/Unbound.html> "Unbound") 有 [unbound-host(1)](<https://man.archlinux.org/man/unbound-host.1>)。
  * [BIND](<../zh-cn/BIND.html> "BIND") 有 [dig(1)](<https://man.archlinux.org/man/dig.1>)、[host(1)](<https://man.archlinux.org/man/host.1>)、[nslookup(1)](<https://man.archlinux.org/man/nslookup.1>) 和一堆 `dnssec-` 工具。
  * [systemd-resolved](<../zh-cn/Systemd-resolved.html> "Systemd-resolved") 有 [resolvectl(1)](<https://man.archlinux.org/man/resolvectl.1>)，它提供了用于DNS 查找的`query`子命令。

##  解析器性能

Glibc 不缓存查询结果。要实现本地缓存，请使用 [systemd-resolved](<../zh-cn/Systemd-resolved.html> "Systemd-resolved") 或者配置一个本地缓存 [DNS 服务器](<#DNS_%E6%9C%8D%E5%8A%A1%E5%99%A8>)将其用作服务器，方法是在 `/etc/resolv.conf`（如果使用 [openresolv](</wzh/index.php?title=Openresolv&action=edit&redlink=1> "Openresolv（页面不存在）")，则在 `/etc/resolvconf.conf`）中设置 `127.0.0.1` 和 `::1` 为名称服务器。 

**提示：**

  * _drill_ 或 _dig_ [查找实用工具](<#%E6%9F%A5%E6%89%BE%E5%AE%9E%E7%94%A8%E5%B7%A5%E5%85%B7>)能报告查询时间。
  * 路由器通常将自己的缓存解析器设置为网络的 DNS 服务器，从而为整个网络提供 DNS 缓存。
  * 如果切换到下一个 DNS 服务器的时间过长，可以尝试[减少超时时间](<#%E9%99%90%E5%88%B6%E6%9F%A5%E6%89%BE%E6%97%B6%E9%97%B4>)。

##  隐私与安全

传统的 DNS 协议是没有加密的，并且不会考虑传输的机密性、完整性和身份验证问题，所以如果您在使用不受信任的网络或者有恶意行为的网络服务供应商（ISP），您的 DNS 查询可能会被窃听并响应[他人的操纵](<https://en.wikipedia.org/wiki/Man-in-the-middle_attack> "wikipedia:Man-in-the-middle attack")。此外，DNS 服务器可进行 [DNS 劫持](<https://en.wikipedia.org/wiki/DNS_hijacking> "wikipedia:DNS hijacking")。 

您需要相信您的 DNS 服务器处理您的查询时能够保密。网络服务供应商和[第三方](<#%E7%AC%AC%E4%B8%89%E6%96%B9_DNS_%E6%9C%8D%E5%8A%A1>)会提供 DNS 服务器。 另外您也可以运营自己的[递归名称服务器](<#DNS_%E6%9C%8D%E5%8A%A1%E5%99%A8>)，但这需要花费更多精力。 如果您在不受信任的网络中使用 [DHCP](</wzh/index.php?title=DHCP&action=edit&redlink=1> "DHCP（页面不存在）") 客户端，请一定要设置静态名称服务器，以避免使用和受制于任意 DNS 服务器。 为了保证您与远程 DNS 服务器的通信安全，您可以使用加密协议，例如 [DNS over TLS](<https://en.wikipedia.org/wiki/DNS_over_TLS> "wikipedia:DNS over TLS")（[RFC 7858](<https://tools.ietf.org/html/rfc7858> "rfc:7858")）、[DNS over HTTPS](<https://en.wikipedia.org/wiki/DNS_over_HTTPS> "wikipedia:DNS over HTTPS")（[RFC 8484](<https://tools.ietf.org/html/rfc8484> "rfc:8484")）或 [DNSCrypt](<https://en.wikipedia.org/wiki/DNSCrypt> "wikipedia:DNSCrypt")，前提是上游服务器和您的解析器都支持该协议。 另一种方法是使用专用软件来加密和解密通信，例如 [stunnel](</wzh/index.php?title=Stunnel&action=edit&redlink=1> "Stunnel（页面不存在）")。 要验证响应是否来自权威的名称服务器，您可以验证 DNSSEC 记录，前提是上游服务器和您的[解析器](<#DNS_%E6%9C%8D%E5%8A%A1%E5%99%A8>)都支持它。 

虽然可以使用加密 DNS 解析器，但浏览器在请求网站证书时仍会泄漏[服务器名称指示（Server Name Indication, SNI）](<https://en.wikipedia.org/wiki/Server_Name_Indication> "wikipedia:Server Name Indication")中的域名。可以使用 [Wireshark](<../zh-cn/Wireshark.html> "Wireshark") 过滤器 `tls.handshake.extensions_server_name_len > 0` 或者下面的命令检查这种泄漏。建议的解决方案是使用[ECH (Encrypted Client Hello)](<https://en.wikipedia.org/wiki/Server_Name_Indication#Encrypted_Client_Hello> "wikipedia:Server Name Indication")，这是 TLS 1.3 协议的扩展。 
    
    # tshark -p -Tfields -e tls.handshake.extensions_server_name -Y 'tls.handshake.extensions_server_name_len>0'
    
###  应用级 DNS

值得注意的是，一些客户端软件，例如主流的网页浏览器[[2]](<https://support.mozilla.org/en-US/kb/firefox-dns-over-https>)[[3]](<https://www.chromium.org/developers/dns-over-https>)，都在开始实现 DNS over HTTPS 功能。虽然一般来说，提供加密查询可以视为一种善举，但这也同时意味着软件可以绕过系统解析器配置，另辟蹊径进行查询。[[4]](<https://blog.powerdns.com/2019/09/25/centralised-doh-is-bad-for-privacy-in-2019-and-beyond/>)

[Firefox](<../zh-cn/Firefox.html> "Firefox") 提供了[配置选项](<https://support.mozilla.org/en-US/kb/firefox-dns-over-https#w_manually-enabling-and-disabling-dns-over-https>)来启用或禁用 DNS over HTTPS 和选择 DNS 服务器。Mozilla 设立了[可信递归解析器 (TRR)](<../zh-cn/Firefox/%E9%9A%90%E7%A7%81.html#%E7%A6%81%E7%94%A8/%E5%BC%BA%E5%88%B6%E4%BD%BF%E7%94%A8%E2%80%9C%E5%8F%97%E4%BF%A1%E9%80%92%E5%BD%92%E8%A7%A3%E6%9E%90%E5%99%A8%E2%80%9D> "Firefox/隐私") 计划，并提供了默认提供商的透明度信息。这里要注意，Firefox 支持并自动为 TRR 提供商启用 ECH，请参阅 [Firefox/隐私#加密客户端问候](<../zh-cn/Firefox/%E9%9A%90%E7%A7%81.html#%E5%8A%A0%E5%AF%86%E5%AE%A2%E6%88%B7%E7%AB%AF%E9%97%AE%E5%80%99> "Firefox/隐私")。 

[Chromium](<../zh-cn/Chromium.html> "Chromium") 会检查用户的系统解析器，如果得知系统解析器地址也提供 DNS over HTTPS 支持，则启用 DNS over HTTPS 功能。请参见[这篇博文](<https://blog.chromium.org/2020/05/a-safer-and-more-private-browsing-DoH.html>)获取更多信息，了解如何禁用 DNS over HTTPS。 

Mozilla [提议](<https://support.mozilla.org/en-US/kb/configuring-networks-disable-dns-over-https>)，如果系统解析器无法解析域名 `use-application-dns.net`，就普遍禁用应用级 DNS。目前，这一功能仅在 Firefox 中实现了。 

###  遗忘式 DNS

遗忘式 DNS（Oblivious DNS，ODNS；[RFC:9230](<https://tools.ietf.org/html/rfc9230> "rfc:9230")）是一套解决了若干 DNS 关切隐私问题的系统。参见 [Cloudflare 的文章（英文）](<https://blog.cloudflare.com/oblivious-dns/>)获取更多信息。

##  DNS 服务器

###  条件转发

在查询特定域名时，可以使用特定的 DNS 解析器。这在连接到虚拟专用网络（VPN）时特别有用，这样，对 VPN 网络的查询由 VPN 的 DNS 解析，而对普通互联网的查询仍由标准 DNS 解析器解析（分流）。这也可以用于本地网络中。 

要实现这个目的，您需要使用[本地解析器](<#DNS_%E6%9C%8D%E5%8A%A1%E5%99%A8>)，因为 glibc 并不支持这样做。 

在动态环境中（笔记本和某种程度的台式机），需要根据所连接的网络来配置解析器。最好的方式是使用 [openresolv](</wzh/index.php?title=Openresolv&action=edit&redlink=1> "Openresolv（页面不存在）")，因为它支持[多个订阅器](</wzh/index.php?title=Openresolv&action=edit&redlink=1> "Openresolv（页面不存在）")。一些[网络管理器](<../zh-cn/Network_manager.html> "Network manager")会通过 openresolv 或者直接通过配置解析器来支持此功能。NetworkManager [支持条件转发而无需 openresolv](<../zh-cn/NetworkManager.html#DNS_caching_and_conditional_forwarding> "NetworkManager")。 

**注意：** 虽然您可以使用其他条件进行转发（比如源 IP 地址），但是“条件转发”一名似乎更用于描述“域名查询”这一条件。
