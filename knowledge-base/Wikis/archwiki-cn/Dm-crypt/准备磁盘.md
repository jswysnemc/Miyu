##  安全擦除驱动器数据

在决定使用何种方式安全擦除驱动器数据时，记住：如果驱动器要持续地作为加密设备使用，这只需要执行一次。 

**警告：**

  * 在开始前对必要数据进行恰当的备份！！
  * 如果您要[使用 TRIM](<../../zh-cn/Dm-crypt/%E7%89%B9%E6%AE%8A%E5%BA%94%E7%94%A8.html#%E5%9B%BA%E6%80%81%E7%A1%AC%E7%9B%98%E7%9A%84Discard/TRIM%E6%94%AF%E6%8C%81> "Dm-crypt/特殊应用")，切勿用随机数据覆写[固态硬盘](<../../zh-cn/%E5%9B%BA%E6%80%81%E7%A1%AC%E7%9B%98.html> "固态硬盘")。第一次执行 TRIM 后，未使用的块将被标记为空，并最终被擦除，使已执行的操作失效。

**注意：** 在清除大量数据时，这个过程会耗费数小时至数天

**提示：**

  * 继续操作前，请检查您的先进格式化设备是否采用[最佳逻辑扇区大小](<../../zh-cn/%E5%85%88%E8%BF%9B%E6%A0%BC%E5%BC%8F%E5%8C%96.html> "先进格式化")。
  * 在数 TB 的设备上，填充加密驱动器的过程可能需要耗费一整天。为了不让机器在操作过程中无法使用，从已经安装在另一个驱动器上的系统中执行此操作，比在 Arch 安装用的 live 系统中执行此操作要好。
  * 对于[固态硬盘](<../../zh-cn/%E5%9B%BA%E6%80%81%E7%A1%AC%E7%9B%98.html> "固态硬盘"),要想尽量减少[闪存](<../../zh-cn/%E5%AE%89%E5%85%A8%E5%9C%B0%E6%93%A6%E9%99%A4%E7%A3%81%E7%9B%98.html#%E9%97%AA%E5%AD%98> "安全地擦除磁盘")的缓存，可以考虑按 [固态硬盘记忆单元清理](</wzh/index.php?title=SSD_memory_cell_clearing&action=edit&redlink=1> "SSD memory cell clearing（页面不存在）") 中的 _优先_ 节执行。

###  通用方案

关于如何擦除并准备好驱动器的详细指导，参考[安全地擦除磁盘#闪存](<../../zh-cn/%E5%AE%89%E5%85%A8%E5%9C%B0%E6%93%A6%E9%99%A4%E7%A3%81%E7%9B%98.html#%E9%97%AA%E5%AD%98> "安全地擦除磁盘")。 

##  dm-crypt 专用方案

下面两个方案专门用于 dm-crypt ，之所以提到它们是因为它们很快而且可以在分区设置后执行。 

[cryptsetup 常见疑问](<https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#2-setup>)(item 2.19 "_How can I wipe a device with crypto-grade randomness?_ ")提到了一个非常简单的操作可以使用一个已存在的 dm-crypt 卷，像一个简单的伪随机数生成器生成随机数据来擦除一个底层块设备上所有可访问的空闲空间。而且据说这可以避免使用方式的暴露。这是因为加密数据实际上与随机数据难以区分。 

####  dm-crypt 擦除空设备或分区

首先，在待加密分区(使用 `sdXY` 的格式)里或在待加密的整个设备(使用 `sdX` 的格式)创建一个临时的加密容器。将使用的密钥从标准 aes-cbc 更改为 aes-xts 可能更好，因为它可能有显著的速度优势(通过 `cryptsetup benchmark` 检查): 
    
     # cryptsetup open --type plain -d /dev/urandom --sector-size 4096 /dev/_block-device_ to_be_wiped
    
您可以这样验证它是否存在: 
    
    # lsblk
    
    NAME          MAJ:MIN RM  SIZE RO TYPE  MOUNTPOINT
    sda             8:0    0  1.8T  0 disk
    └─to_be_wiped 252:0    0  1.8T  0 crypt
    
用0擦除容器。不需要使用 `if=/dev/urandom` 因为加密密钥已经将数据变成随机的了。 
    
    # dd if=/dev/zero of=/dev/mapper/to_be_wiped status=progress bs=1M
    
    dd: writing to ‘/dev/mapper/to_be_wiped’: No space left on device

**提示：** 要想对操作进行检查,在创建擦除容器前将分区清零。在执行擦除命令后 `blockdev --getsize64 _/dev/mapper/container_` 可以作为 root 获取准确的容器大小。在容器被关闭后 _od_ 可以用于检查擦除操作是否覆盖了被清零的分区。举个例子,执行命令 `# od -j $((_containersize - blocksize_)) /dev/nvme0n1` 来从头至尾检查擦除是否执行。

最后，关闭临时的容器: 
    
    # cryptsetup close to_be_wiped
    
在加密整个系统时，下一步是[#分区](<#%E5%88%86%E5%8C%BA>)。如果只是加密一个分区，继续按照[加密非 root 文件系统#分区](<../../zh-cn/Dm-crypt/Encrypting_a_non-root_file_system.html#Partition> "Dm-crypt/Encrypting a non-root file system")执行。 

####  dm-crypt 在安装后擦除可用空间

在[安装](<../../zh-cn/Dm-crypt/%E5%8A%A0%E5%AF%86%E6%95%B4%E4%B8%AA%E7%B3%BB%E7%BB%9F.html> "Dm-crypt/加密整个系统")前没时间执行擦除步骤的用户，即使在加密系统已启动，文件系统已挂载的情况下也可以做到类似的效果。但请注意相关文件系统是否有保留空间，例如，对于 root 用户，或即使是 root 用户执行的擦除操作也会被限制的其他[磁盘配额](</wzh/index.php?title=Disk_quota&action=edit&redlink=1> "Disk quota（页面不存在）")机制:底层块设备的一些部分可能完全不会写入。 

要想执行擦除操作，通过在加密容器写入一个文件来临时填充分区的剩余可用空间: 
    
    # dd if=/dev/zero of=/file/in/container status=progress
    
    dd: writing to ‘/file/in/container’: No space left on device

缓存同步到磁盘，然后删除文件来回收可用空间。 
    
    # sync
    # rm /file/in/container
    
对每个块设备的分区和其中的文件系统都要重复一次以上操作。举个例子，设置 [LUKS 上的 LVM](<../../zh-cn/Dm-crypt/%E5%8A%A0%E5%AF%86%E6%95%B4%E4%B8%AA%E7%B3%BB%E7%BB%9F.html#LVM_on_LUKS> "Dm-crypt/加密整个系统") 时，对每个逻辑卷都要执行这个操作。 

####  dm-crypt 在安装后擦除可用空间(通过重新加密)

作为替代，对于想要在不重新安装的情况下完全擦除可用空间的用户，可以通过[重新加密 LUKS 设备](<../../zh-cn/Dm-crypt/%E8%AE%BE%E5%A4%87%E5%8A%A0%E5%AF%86.html#%E9%87%8D%E6%96%B0%E5%8A%A0%E5%AF%86%E8%AE%BE%E5%A4%87> "Dm-crypt/设备加密")来达成。需要在每个 LUKS 设备上执行一次。但是请注意这个过程可能很慢(在桌面 HDD 上大约 50MB/s)。 

###  擦除 LUKS header

使用 LUKS 的分区由两部分组成：标头和加密数据。标头包含密钥和加密参数，如果没有这些密钥和参数，几乎不可能恢复数据。如果创建新分区或停用硬盘，只需删除该部分而无需擦除整个设备。注意事项参见本小节末。 

**警告：** 操作是不可逆的，请小心执行。

要想擦除所有密钥，使用下列命令: 
    
    # cryptsetup erase _device_
    
通过以下命令确保没有激活的密钥槽: 
    
    # cryptsetup luksDump _device_
    
此外，通过在密钥被擦除后调用 [wipefs](</wzh/index.php?title=Wipefs&action=edit&redlink=1> "Wipefs（页面不存在）") 可以移除 LUKS header 本身以避免 `cryptsetup` 之后检测到它: 
    
    # wipefs -a _device_
    
加密数据保留在原位。截至2020年，没有已知的实用方法解密这些数据，但是在未来可能会有所不同。由用户来衡量安全，隐私与正确擦除整个设备的时间。 

在某些存储介质（尤其是闪存介质）上，可能无法可靠地覆盖数据。带有密钥的 LUKS 标头可能会保存在操作系统无法访问的位置（如驱动器缓存）。如果担心出现这种情况，则必须使用[ATA 安全擦除](<../../zh-cn/%E5%AE%89%E5%85%A8%E5%9C%B0%E6%93%A6%E9%99%A4%E7%A3%81%E7%9B%98.html#hdparm> "安全地擦除磁盘")。该操作会擦除设备上的所有区块，包括软件看不到的区块。详情请参见 [cryptsetup FAQ 5.19](<https://gitlab.com/cryptsetup/cryptsetup/wikis/FrequentlyAskedQuestions#5-security-aspects>)。 

##  分区

**注意：** 如果您要使用[OPAL 硬件加密](</wzh/index.php?title=Self-encrypting_drives&action=edit&redlink=1> "Self-encrypting drives（页面不存在）")，且之前未为硬盘设置 OPAL 管理密码，则需要使用 [cryptsetup-erase(8)](<https://man.archlinux.org/man/cryptsetup-erase.8>) 删除硬盘上的所有数据，才能创建新的 OPAL 管理密码。请按照[Self-encrypting drives#Secure disk erasure](</wzh/index.php?title=Self-encrypting_drives&action=edit&redlink=1> "Self-encrypting drives（页面不存在）")中的说明，在分区前执行擦除操作。

这一节只有在加密整个系统才需要。在驱动器被安全地覆写后，必须慎重地选择一个恰当的分区方案，考虑到 dm-crypt 的要求，以及各种选择对最终系统管理的影响。 

值得注意的是，从现在起在几乎所有情况下都必须有一个用于 `/boot` 的单独保持未加密分区，因为引导加载器需要访问用于加载 initramfs/encryption 模块的 `/boot` 路径，以此加载剩余的系统(详情请见 [mkinitcpio](<../../zh-cn/Mkinitcpio.html> "Mkinitcpio"))。如果这会产生安全问题，请见 [dm-crypt/特殊应用#保护未加密的boot分区](<../../zh-cn/Dm-crypt/%E7%89%B9%E6%AE%8A%E5%BA%94%E7%94%A8.html#%E4%BF%9D%E6%8A%A4%E6%9C%AA%E5%8A%A0%E5%AF%86%E7%9A%84boot%E5%88%86%E5%8C%BA> "Dm-crypt/特殊应用")。 

另一个需要考虑的重要因素是如何处理交换区和系统暂停，请见 [dm-crypt/交换分区加密](<../../zh-cn/Dm-crypt/%E4%BA%A4%E6%8D%A2%E5%88%86%E5%8C%BA%E5%8A%A0%E5%AF%86.html> "Dm-crypt/交换分区加密")。 

###  物理分区

在最简单的情况下，加密层可以直接基于物理分区;对于如何创建物理分区，请见[分区](<../../zh-cn/%E5%88%86%E5%8C%BA.html> "Partitioning")。就像在未加密系统中，一个 root 分区就足够了，此外还需要一个上述的 `/boot` 分区。使用这个方法可以选择哪些分区加密，哪些分区不加密，无论涉及的磁盘数量多少，都可以正常工作。之后还可以添加或移除分区，但是调整分区的大小会被分区所基于的磁盘限制。最后请注意每个加密分区的开启都需要单独的密钥，即使这可以通过使用 `crypttab` 文件在启动时自动化，请见 [Dm-crypt/系统配置#crypttab](<../../zh-cn/Dm-crypt/%E7%B3%BB%E7%BB%9F%E9%85%8D%E7%BD%AE.html#crypttab> "Dm-crypt/系统配置")。 

###  堆叠块设备

如果需要更高的灵活性，dm-crypt 可以和其他例如 [LVM](<../../zh-cn/LVM.html> "LVM") 和 [RAID](<../../zh-cn/RAID.html> "RAID")的堆叠块设备共存。加密容器可以位于其他堆叠块设备的之下或基于其他堆叠块设备上: 

  * 如果 LVM/RAID 设备创建在加密层上，那么在同一加密分区中可以自由地添加，移除文件系统和调整文件系统大小，所有的存储设备都只需要一个密钥或密码。因为加密层基于物理分区，但是这样就不能使用 LVM 和 RAID 跨越多个磁盘的能力。
  * 如果加密层创建在 LVM/RAID 设备上，它之后依然有可能重组文件系统，但是复杂性会提高，因为加密层需要相应地调整。而且，每个加密设备的打开都需要一个密钥或密码。不过这是在跨越多个磁盘上创建加密文件系统的唯一选择。
